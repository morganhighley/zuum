Driver SDK V2 (Entity Model) / SDK Framework / Configuration Flow

Configuration Flow

When a driver is loaded, it typically requires some configuration to connect to the device. This includes items such as IP addresses, COM Ports, authentication credentials, PIN codes, options, message boxes, and previously‑saved driver information. Historically, specific C# interfaces were used to configure communications, and User Attributes were used for custom items.

In Entity Model drivers, a unified configuration flow covers both use cases. A descriptor named UsageContext indicates how the property is used so that the application can provide a tailored experience for something like COM Port selection. However, the items are all self-describing so that if in the future a new transport type is added that only requires data (as opposed to hardware resources), it can still be loaded and configured on a system that predates the addition of that type.

There are three primary use cases in the configuration flow which are described in the following sections.

Initial Configuration

When the driver is first added to the system, it needs to be configured. The installer is interacting with the system and must be walked through this process. The driver presents its configuration items to the application grouped in sequential steps, one step at a time. The driver can choose the next step dynamically based on responses in the previous steps, and can asynchronously update items in the present step to change their values or descriptions. The driver can also report errors for any invalid items received.

When the driver is fully configured, it returns null for the next step ID.

NOTE: Some systems may have use cases where it is desirable to perform configuration before the driver is loaded. Where possible, the DAT file should be populated with the required configuration items for this to occur.

Apply a Saved Configuration

After a driver has been configured, either by following the flow described in Initial Configuration in a previous boot or by collecting items from the DAT file, the application has all of the configuration values ready at once. In this situation, the application will apply all of the configuration values (from all steps) at once. This is an all or nothing action: if not all values are present, the driver is not expected to remember the ones that were there previously.

Typically, there should be no errors at this step as the configuration flow should have been completed successfully in a previous boot. However, the driver may report errors back to the application if they occur. In this case, the driver would remain offline.

Another use case for applying a saved configuration is when the installer edits a configuration item while the driver is running. In this case, the driver may choose to either completely reinitiailize with the new settings, or determine whether that action is necessary based on what changed.

NOTE: Even if only one setting is changed, the application is required to send the full set of items.

Updating a Configuration Item Value During Operation (Saved Driver Settings)

Sometimes, a driver will need to persist information it acquires and updates during operation. For example, an access token may be refreshed, or a user's preference may need to be remembered.

In these cases, a UsageContext of Data can be used to specify an item which is not shown to the user and is passed back to the driver on the next boot. The ConfigurationItemsUpdated event can be used to update the value. To raise this event in a driver using the DelegateDataDrivenConfigurationController class, call DelegateDataDrivenConfigurationController.NotifyValuesChanged() with the updated values.

NOTE: This event works even for other UsageContext values, which may be useful if the user can specify a default while configuring.

Implement the Configuration Flow

In most cases, the workflow described in Add and Define Configuration Items is the best way to implement the configuration flow. The JSON file defines the items for the SDK to utilize in its state machine, calling back into the driver to apply the item values.

However, when the driver wants to retain total control, it can manually implement IDriverConfigurationController. Refer to the preceding diagrams for a visual depiction of how this interface operates.

At any time, the application may call GetStatus() to check if the driver is fully configured (DriverControllerStatus.Running) or not configured yet (DriverControllerStatus.NotConfigured). Other statuses should not be used by the driver. Any time the status changes (even if in response to an API call by the application, such as ApplyConfiguration), the driver must raise the StatusChanged event.

The application may call SetTransportImplementation before or during configuration, or not at all if not supported by the application. This method provides a delegate that the driver may use to send data out of a COM, CEC, or similar port. It also allows delegates to subscribe and unsubscribe to incoming data for a particular channel ID. When the driver requires a communications channel such as a serial port, it must include a configuration item with the appropriate UsageContext. The application will provide a string ID for that item that can be used with the delegates here to communicate via the port. UnsetTransportImplementation may be called by the application to disconnect this transport, and in response, the driver should unsubscribe from any incoming data channels and perform any relevant cleanup. If the driver does not need any of these communications channels, it can do nothing in its implementation of these methods.

The application will configure the driver in one of two ways. When the set of configuration values is already known (from a previous configuration or from user input based on .dat file contents), the application may call ApplyConfiguration with the complete set of values. The driver must accept these and return a result describing any errors that caused the driver to remain unconfigured. When the set of configuration items is not known yet, the application may call GetFirstConfigurationStep to retrieve the initial set of items. After the user inputs the required items, the application will call ApplyConfigurationStep to submit those to the driver and retrieve the next step. This process continues until all items are populated and the driver returns null for the next step. In either of these cases, the driver must raise the proper events for the item value and status changes.

Once the driver is configured, the application may still call ApplyConfiguration with updated values at any time if the user changes a setting declared with the Always availability. The application may provide all values or only the changed ones.
