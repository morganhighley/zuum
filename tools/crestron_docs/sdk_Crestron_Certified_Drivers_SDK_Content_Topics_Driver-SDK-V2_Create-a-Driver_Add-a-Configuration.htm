Add and Define Configuration Items

In order to add configuration items to the driver, they must first be added to the JSON file created in Create Required Components. In addition to powering the helper classes in the SDK, this tells ManifestUtil what to put into the DAT file.

The following example uses a light driver with an IP configuration as an example. Copy in the configuration items for an IP transport as shown below, editing the DefaultValue of the _Port_ ID to match your device. For other communication types, copy their relevant items instead as described in Configuration Item Definitions.

Copy 
{
    "ConfigurationSteps": {
        "Items": [
            {
                "Id": "_Host_",
                "UsageContext": "Network.Address",
                "Title": "IP Address / Hostname",
                "Required": true,
                "Persistent": true,
                "ValueType": "String",
            },
            {
                "Id": "_Port_",
                "UsageContext": "Network.Port",
                "Title": "Port",
                "Required": true,
                "MaxValue": 65535,
                "MinValue": 1,
                "Persistent": true,
                "ValueType": "Number",
                "DefaultValue": "443"
            }
        ],
        "Steps": [
            {
                "StepId": "Transport",
                "Items": [
                    "_Host_",
                    "_Port_"
                ],
                "NextStep": null
            }
        ],
        "FirstStep": "Transport"
    }
}

Next, write a handler in the driver for these items. The Driver SDK will handle validation of the data types, ranges, required state, and so forth for the values. The driver must validate that the host is valid, save the values, and start communicating with the device.

This validation is accomplished via a delegate in the driver that is called whenever one of the following scenarios occurs:

  • ApplyConfiguration is called with all valid items, such as for reloaded saved configuration values after an application restart. In these cases, action will be ApplyAll and stepId will be null.
  • ApplyConfiguration is called with one or more updated items after the driver is already configured, but not necessarily with all items. In this case, the driver may still return errors for invalid settings or combinations of settings. action will be ApplyAll and stepId will be null (even if not all items are updated).
  • ApplyConfigurationStep is called with valid items for a given step. In this case, action will be ApplyStep and stepId will be the ID associated with the step.
  • The application is going back (for example, ApplyConfigurationStep is called for a previous step), and the driver needs to clear any configuration values that were backed out. In this case, action will be ClearValues and stepId will be null.

Add the following to MyDriverEntity:

Copy 
using Crestron.DeviceDrivers.EntityModel;
using Crestron.DeviceDrivers.EntityModel.Data;
using Crestron.DeviceDrivers.SDK;
using Crestron.DeviceDrivers.SDK.EntityModel;
using System.Collections.Generic;

public class MyDriverEntity : ReflectedAttributeDriverEntity
{
    public MyDriverEntity(DriverControllerCreationArgs args, DriverImplementationResources resources) : base(DriverController.RootControllerId)
    {
        DataDrivenConfigurationControllerArgs cfgArgs = DataDrivenConfigurationControllerArgs.FromResources(args, resources, DriverController.RootControllerId);
        ConfigurationController = new DelegateDataDrivenConfigurationController(cfgArgs, ApplyConfigurationItems, null);
    }

    internal DataDrivenConfigurationController ConfigurationController { get; private set; }

    private ConfigurationItemErrors ApplyConfigurationItems(
        DataDrivenConfigurationController.ApplyConfigurationAction action,
        string stepId,
        IDictionary<string, DriverEntityValue?> values
    )
    {
        switch (action)
        {
            // In this case, we only have one step, so these are the same
            case DataDrivenConfigurationController.ApplyConfigurationAction.ApplyAll:
            case DataDrivenConfigurationController.ApplyConfigurationAction.ApplyStep:
                var host = values["_Host_"].Value.GetValue<string>();
                var port = (ushort)values["_Port_"].Value.GetValue<long>();
                // TODO: Validate host, create transport (in a real driver)
                // If host is invalid...
                // return new ConfigurationItemErrors(new Dictionary<string, string> { { "_Host_", "Invalid host name" } }, null);
                return null;

            case DataDrivenConfigurationController.ApplyConfigurationAction.ClearValues:
                // This happens when the user goes "back" to a previous step during configuration.
                // In our use case here, this is impossible since there's only one step.
                // This code is just provided for reference.
                // The dictionary of values will have items to clear.
                if (values.ContainsKey("_Host_"))
                {
                    // Destroy transport, await new values
                }
                break;
        }

        return null;
    }
}

In order to expose this configuration to the application, pass the configuration controller to ConfigurableDriverEntity.

Edit EntryPoint.CreateDriverControllerInstance() as needed:

Copy 
var entity = new ConfigurableDriverEntity(driverEntity.ControllerId, driverEntity, driverEntity.ConfigurationController);

At this point, the driver is now loadable and configurable, but it still does not support any properties or commands. These items will be added in later topics.

Update Configuration Items

Both the values and descriptions of configuration items can be updated during the configuration flow or during driver operation. Values are most often updated within items with a UsageContext of Data for an invisible configuration item that the application saves and then gives back to the driver when it is restarted. This can be used to save authentication tokens and similar values.

Descriptions are most often updated when displaying instructions or verification codes. However, some applications will attempt to read configuration items out of the driver’s .dat file and present the user with a static set of options. When that approach is not viable for the driver, the JSON ConfigurationSteps must set IsNotOfflineConfigurable to true so that the application does not attempt this approach. Alternatively, when it is possible to configure offline with alternate instructions, the items may include a NonInteractiveDescription, which is used in place of Description when generating the .dat file (and similarly, a NonInteractiveTitle).

To implement complete control over the configuration process, refer to Configuration Flow.

When using the DataDrivenConfigurationController, the NotifyValuesChanged method is used to update configuration items. The value is updated by providing a dictionary key matching the item Id. Updates to descriptions are provided using placeholders in template strings. For example, a description could be “Enter the following PIN Code: {PinCode}”, and the driver can call NotifyValuesChanged with a key of “PinCode” to update and fill in that value. Alternatively, the description could be “{MyDescription}” and then the key “MyDescription” could be used to provide the entire description value from the driver.

Configure Child Devices

Drivers that support the platform:managedDevices property and include child devices may optionally define configuration items for those child devices. DispatchingDeviceController expects a ConfigurableDriverEntity instance per child device, and the constructor to that class takes an IDriverConfigurationController parameter, which may be null if no configuration is needed.

While the JSON file does not itself support configurations for child devices, the same infrastructure may be used in C# to instantiate a DelegateDataDrivenConfigurationController to handle child device configuration. Refer to Configuration Item Definitions for details on what data may be defined. The sample code below shows how to define such items in C# for a child device. A similar approach could be used to dynamically populate this data for the root device as well.

Copy 
// From existing sample code at initialization - some items from this class should be saved off to reuse 
// Specifically, the ConditionLookup, TransformationLookup, and Logger will be needed for the child device configuration. 
DataDrivenConfigurationControllerArgs cfgArgs = DataDrivenConfigurationControllerArgs.FromResources(args, resources, DriverController.RootControllerId);

// This string must match the key in the platform:managedDevices dictionary 
string childControllerId = "my-child-01";

// This string should match the model specified in platform:managedDevices for this child 
string childDeviceModel = "MyChildDeviceModel";

// This is the same data that is normally defined in JSON 
ConfigurationStepsDefinition childConfigurationSteps = new ConfigurationStepsDefinition
{
    Items =
    {
        new ConfigurationItemDefinition
        {
            Id = "ChildItem1",
            Title = "Title of the child item",
            Description = "Description of the item",
            Availability = ConfigurationItemAvailability.Always,
            ValueType = ConfigurationItemValueType.String,
            UsageContext = ConfigurationItemContext.Generic.Prompt,
            DefaultValue = "The default value",
            Required = true,
            Persistent = true,
        },
    },
    Steps =
    {
        new ConfigurationStepDefinition
        {
            StepId = "ChildStep1",
            Items =
            {
                "ChildItem1",
            },
        },
    },
    FirstStep = "ChildStep1",
    IncludePersistentValueData = false,
    IsNotOfflineConfigurable = false,
};

// Later, when creating a child device which requires configuration 
DataDrivenConfigurationControllerArgs childCfgArgs = new DataDrivenConfigurationControllerArgs(
    childDeviceModel,
    childControllerId,
    childConfigurationSteps,
    // This is only used for internal SDK infrastructure. Leave it as this value. 
    Enumerable.Empty<KeyValuePair<string, IList<ConfigurationItemDefinition>>>(),
    // Pass these along from the original configuration args. 
    // If unavailable, empty delegates which return null may be used (_ => null) 
    cfgArgs.ConditionLookup,
    cfgArgs.TransformationLookup,
    // Optionally: create a dedicated component logger instead of reusing the logger for the parent configuration 
    // resources.Logger.GetComponentLogger(LoggingConstants.ComponentTypeOther, childControllerId), 
    // If necessary, a custom implementation of IComponentLogger may be passed in here, even it the methods are stubbed out no-ops. 
    cfgArgs.Logger
);

// Here, child.ApplyConfigurationItems is the driver-defined delegate and should point to whatever 
// method handles applying child configuration items 
IDriverConfigurationController childConfigurationController = new DelegateDataDrivenConfigurationController(childCfgArgs, child.ApplyConfigurationItems);
