Driver SDK V2 (Entity Model) / Create a Driver / Add Extension Support

Add Extension Support

The Crestron Home Extension feature enables a driver developer to present a custom user interface in Crestron Home® OS. Any Driver SDK V2 (Entity Model) device type may support the Crestron Home Extension feature, including both native device types and nonnative device types as described in Supported Device Types. For the Extension aspects of the device, the driver developer decides how the end user interacts with the device via their custom user interface. The custom user interface is defined by the UI definition, which is an XML file containing the layout of the UI.

In addition to the standard .dat and .dll files, a driver supporting the Crestron Home Extension feature contains additional components inside the .pkg file that make up the Crestron Home Extension driver:

  • A UiDefinitions folder containing the UI definition file for the Crestron Home Extension driver
  • A Translations folder containing the translation files for the Crestron Home Extension driver

NOTE: Definition and translation data can also be implemented dynamically instead of through the files described above. For simplicity in this documentation, all implemented definition and translation data will be referred to as "files".

Crestron Home Extension Device Components

A Crestron Home Extension device is made up of the following components:

  • UI Definition
  • Language Translations
  • Properties (with the metadata value ExtensionUiProperty = true)
  • Commands

Using the UI definition, translations, and properties, Crestron Home can send user actions to your device driver via controls such as buttons and text entry boxes. These actions are called on your driver via the commands specified in the UI Definition file.

Each component that makes up a Crestron Home Extension device is described in the sections that follow.

UI Definition

The purpose of the UI definition is to define the layout of the UI that the end user will interact with through the Crestron Home app. The UI definition also defines the commands that are sent from the UI to the driver when certain actions are performed, such as a button press. The UI definition can use keys from the language translations to display static text or it can use the property values to display information from the device itself. For example, you can have a property named “PowerState” that has the value “On” or “Off,” which can then be displayed in the UI.

For more information on creating a UI definition file, refer to UI Definition.

Language Translations

Language translations provide a way for your driver to be used in any language that the developer decides to support. They are written as key/value pairs, where the keys are referenced by the UI definition and the values are the translated strings.

For more information on creating language translations, refer to Translation Files.

If predefined translation file sets do not meet the driver’s needs, then translations may be implemented dynamically by the driver instead. To accomplish this, the DriverAssemblyEntryPoint.CreateDriverControllerInstance method must provide its own IDriverControllerLocalization implementation instead of null when creating the DispatchingDeviceController instance.

NOTE: If DispatchingDeviceController is not being used, custom implementations of the localization‑related members of DriverController should be used instead.

This interface must return an immutable list of supported cultures (raising an event whenever that changes) and implement a method to retrieve an immutable dictionary of translations for a given culture string (raising an event whenever that dictionary should be retrieved again due to changes). The cultures and dictionaries follow the same rules as the translation files but are loaded dynamically. Refer to the C# API documentation for further information on each member that must be implemented.

IncludeInPkg Folder

Crestron Home Extension devices require the UI definition and language translations to be included in the driver’s PKG file so that consuming applications such as Crestron Home can use them.

Including these files within the PKG file requires placing them in a folder named IncludeInPkg, which is provided to the Manifest Utility tool along with the driver’s DLL file in order to create the PKG. Any files or folders within the IncludeInPkg folder will be included in the driver’s PKG file.

To create and populate the IncludeInPkg folder:

  1. Create a folder named IncludeInPkg.
  2. Create two subfolders named UiDefinitions and Translations. These subfolder names must be entered exactly as shown.
  3. Place your UI definition file within the UiDefinitions folder.
  4. Place your translation files within the Translations folder.
  5. Include the IncludeInPkg folder in the PKG file as described in Build the Driver.

Implement Properties and Commands for UI Definition

Commands and properties used in the UI Definition are defined in the same way as other commands and properties in the driver. However, there are some important differences:

  • Extension UI properties and commands must not use a namespace: prefix. For example, use vacuumStart instead of vacuum:start.

    • Namespaced properties are reserved for those defined by Crestron.

    • The UI Definition schema does not allow for special characters in a name.

    Properties used in the UI Definition must have ExtensionUiProperty = true in their metadata.

    • This setting prevents unwanted or unused properties from being sent to the device hosting the UI..

    • This can be done by adding the [EntityPropertyMetadata(ExtensionUiProperty = true)] attribute to a property that already has the [EntityProperty] attribute.

    • Custom types (classes) in the driver’s property data structures must derive from ExtensionObject, which provides the ExtensionObjectId property in order to identify the object when a property on it is set.

  • Some initial setup is required in the driver to enable binding to the driver’s commands.

UI Bindings

For specifics on the syntax required in the UI Definition to bind to a command or a property, refer to UI Definition.

Crestron Home Extensions predate the Driver SDK V2 (Entity Model), so some adapter code is required to bridge the UI Definition and implementation to the Entity Model APIs. The UI expects the following components:

  • Language translations

  • A string property named extension:uiDefinition containing the xml content

  • A command named extension:doCommand with one string parameter named commandName and a string array (string[]) parameter named args

    • This command is called when referenced from the UI Definition and the relevant action occurs (for example, a button is pressed).

  • A command named extension:setPropertyValue with three string parameters named objectId, property, and value, respectively

    • This command is called whenever a control on the UI changes the value of the property it is bound to in the UI Definition (for example, when the user moves a slider).

To provide the necessary components for Extension functionality, the driver should include the property and command implementations available in the SDK. In the constructor of your driver entity class, add the following:

Copy 
var uiDefinition = UiDefinitionProperty.LoadFromDirectoryIfExists(args.DriverDataDirectoryPath, resources.InitLogger, LogEntryLevel.Error); 
AddProperty(uiDefinition, UiDefinitionProperty.Name, uiDefinition);

var doCommand = new ExtensionDoCommandExecutor(GetCommand); 
AddCommand(this, ExtensionDoCommandExecutor.CommandName, doCommand); 

var setPropertyValue = new ExtensionSetPropertyValueExecutor(GetCommand); 
AddCommand(this, ExtensionSetPropertyValueExecutor.CommandName, setPropertyValue);

Language translations are automatically populated by the SDK when you include them in the Translations directory of the PKG file. Similarly, adding the UiDefinitionProperty above implements extension:uiDefinition by locating and reading  UiDefinitions/UiDefinition.xml in the PKG file.

UiDefinitionProperty.LoadFromDirectoryIfExists will load the first XML file it finds in the UiDefinitions directory. If the driver needs to support multiple different XML files, it can use the constructor for UiDefinitionProperty to specify an absolute path to a specific file. This is most commonly used in a Platform driver with multiple child devices that have their own XML files.

If the UI Definition is changed, UiDefinitionProperty.GetValue can be called to read the new file contents. This can be then used with NotifyPropertyChanged(UiDefinitionProperty.Name, value) to notify the application of the update.

Alternatively, the driver can declare a string property extension:uiDefinition manually and populate the XML content with a custom implementation. In that case, notifying the application of updates works just like any other property.

Properties with the ExtensionUiProperty = true metadata value are automatically available to the UI Definition.

To satisfy the final two components (extension:doCommand and extension:setPropertyValue), the executors above will call matching Entity Model commands, converting the parameters to the types specified in the signature.

For extension:doCommand commands, defining commands is enough to make them available to extensions via the ExtensionDoCommandExecutor added above. If they need to return a result code, they should return ExtensionOperationResult.

Refer to the following sample code.

Copy 
[EntityCommand(Id = "myCommand")] 
public void MyCommand() 
{ 
} 

[EntityCommand(Id = "otherCommand")] 
public ExtensionOperationResult OtherCommand(string someArg) 
{ 
    return OperationResult.Success; 
} 

Finally, for extension:setPropertyValue, the ExtensionSetPropertyValueExecutor looks for commands you define matching the following criteria:

  • The command name is the capitalized property name prefixed with “set”. For example, a property named “activeLightShow” requires its command to be named “SetActiveLightShow”.

  • If the property is on an object, it should take a string parameter named “objectId” as its first parameter.

  • The command must take a string parameter named “value”.

Two examples are listed below. The comments indicate the equivalent operations from the Driver SDK V1 (RAD Framework).

Copy 
// Can be invoked by equivalent of DoCommand("setActiveLightShow", new string[] { "the show" }) 
// Can be invoked by equivalent of SetPropertyValue(null, "activeLightShow", "the show") 
// Can optionally return ExtensionOperationResult to set success / error message 
[EntityCommand(Id = "setActiveLightShow")] 
public void SetActiveLightShow(string value) 
{ 
    ActiveLightShow = value; 
    NotifyPropertyChanged("activeLightShow", new DriverEntityValue(value)); 
} 
 
// Can optionally return ExtensionOperationResult to set success / error message 
[EntityCommand(Id = "setMyData")] 
public void SetMyData(string objectId, int value) 
{ 
    MyCustomData instance = PropWithCustomData.FirstOrDefault(x => x.ExtensionObjectId == objectId); 
    if (instance != null && instance.MyData != value) 
    { 
        instance.MyData = value; 
        NotifyPropertyChanged("propWithCustomData", new DriverEntityValue(PropWithCustomData)); 
    } 
} 

public class MyCustomData : ExtensionObject 
{ 
    public MyCustomData(int myData) : base() 
    { 
        MyData = myData; 
    } 
 
    [EntityProperty] 
    public int MyData { get; set; } 
} 

// Since no Id is specified, property is is converted to camelCase and will 
// be available in extensions as activeLightShow 
// Note that you can still specify an Id if you want to use that instead 
[EntityProperty] 
[EntityPropertyMetadata(ExtensionUiProperty = true)] 
public string ActiveLightShow { get; private set; } 
 
[EntityProperty] 
[EntityPropertyMetadata(ExtensionUiProperty = true)] 
public MyCustomData[] PropWithCustomData { get; private set; } 

Port SDK V1 (RAD Framework) Extensions

Most of the time, the utilities described earlier in this topic provide a better developer experience and will be preferable when creating a Crestron Home Extension driver. However, it is possible to manually implement extension:uiDefinition, extension:doCommand, and extension:setPropertyValue. An example is provided below.

Copy 
// DO NOT add [EntityPropertyMetadata(ExtensionUiProperty = true)] here 
// This property is extension-related, but it’s not a UI property to be displayed within the UI definition itself 

[EntityProperty(Id = "extension:uiDefinition")] 

public string ActiveLightShow { get; private set; } 

[EntityCommand(Id = "extension:doCommand")] 
public ExtensionOperationResult DoCommand(string commandName, string[] args) 
{ 
    if (commandName == "setActiveLightShow" && ActiveLightShow != args[1]) 
    { 
        ActiveLightShow = args[1]; 
        NotifyPropertyChanged("activeLightShow", new DriverEntityValue(args[1])); 
    } 
 
    return OperationResult.Success; 
} 
 
[EntityCommand(Id = "extension:setPropertyValue")] 
public ExtensionOperationResult SetPropertyValue(string objectId, string property, string value) 
{ 
    if (property == "activeLightShow" && ActiveLightShow != value) 
    { 
        ActiveLightShow = value; 
        NotifyPropertyChanged("activeLightShow", new DriverEntityValue(value)); 
    } 
 
    if (property == "myData") 
    { 
        int value = int.Parse(value); 
        MyCustomData instance = PropWithCustomData.FirstOrDefault(x => x.ExtensionObjectId == objectId); 
        if (instance != null && instance.MyData != value) 
        { 
            instance.MyData = value; 
            NotifyPropertyChanged("propWithCustomData", new DriverEntityValue(PropWithCustomData)); 
        } 
    } 
 
    return OperationResult.Success; 
} 

Build the Driver

Once your driver code is complete, you must build it into a PKG file so that Crestron Home can consume it.

To build a Crestron Home Extension device PKG file:

  1. Build the driver solution.
  2. Copy the DLL file from the output to the folder containing the Manifest Utility tool provided with the SDK. Reference the README file in this folder for additional information on this tool.
  3. Copy the IncludeInPkg folder to the same folder mentioned in step 2. Refer to IncludeInPkg Folder for more information.
  4. Run the Manifest Utility tool by double clicking ManifestUtil.exe

If the build is successful, a PKG file with the same name as the DLL will be created.

NOTE: Do not change the name of the PKG file. It must match the DLL file.

To verify the contents of a PKG file, you can unzip it using 7-Zip or similar utility.

For Crestron Home Extension devices, there should be at least four items in the package contents:

  • The DLL file
  • A DAT file containing a subset of the information from the JSON file.
  • A folder named UiDefinitions containing the UI definition provided.
  • A folder named Translations containing the translation files provided.
