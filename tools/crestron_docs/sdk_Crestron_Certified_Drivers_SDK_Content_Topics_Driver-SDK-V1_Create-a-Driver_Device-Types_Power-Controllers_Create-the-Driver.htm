Create the Driver

The topics in this section describe how to create a power controller driver. This tutorial contains incomplete examples that are used as a guide. For the complete sample code, refer to the Power Controller Sample driver provided in the Crestron Drivers SDK, which can be downloaded from Resources.

Creating a power controller driver requires the following steps:

  1. Create the Driver Project
  2. Create the Data File
  3. Implement the Main Driver Class
  4. Implement the Protocol Class
  5. Implement the Transport Class
  6. Build the Driver PKG File
  7. Test the Driver

Create the Driver Project

Create a new driver project in Visual Studio software as described in Create the Driver Project.

Create the Data File

The driver data file is a JSON-formatted file that contains information about the device, and it is embedded into the DLL file. Since the data file format is the same across all device types, only some of the information in a data file applies to power controller devices.

To create a data file for a power controller device, modify the example given below with the details about your device.

NOTE: For power controller drivers, the value of the DeviceType field must be “Power Controller” and the minimum SdkVersion must be set to “15.00.000”. This is the version where support for power controller drivers was added.

After the data file is created, it will need to be imported into the driver’s project and set as an embedded resource.

To add an existing JSON file to the project:

  1. Right click on the project in Visual Studio.
  2. Navigate to Add > Existing Item.
  3. Select the JSON file.
  4. Select Add.

To set the JSON file as an embedded resource, navigate to the properties of the file in Visual Studio, and for Build Action, select Embedded Resource.

Implement the Main Driver Class

Create a new class that implements ABasicDriver and the transport interface being used (ITcp, ICloudConnected, and so forth). In the Initialize method, set the transport and the protocol. The protocol class will be created in the next step. Power controllers must also implement IPowerBankContainer and IReadyIndicator.

The following example code uses the ITcp transport interface. For the complete sample code, refer to the Power Controller Sample driver provided in the Crestron Drivers SDK.

Implement Device Capabilities

Capabilities can be added to your power controller driver depending on the functions that the device supports. A capability is any class that implements IDeviceCapability. For more information on the capabilities supported by power controllers, refer to Overview.

There are two methods that can be used to implement capabilities. The examples below show each method in conjunction with implementing IPowerBankCapability, though they apply to any capability.

  1. The interface can be implemented directly onto your driver class.

    2. Copy 
    public class SampleDriver : ABasicDriver, ITcp, IPowerBankContainer, IReadyIndicator 
    { 
        ...

        public bool IsReady { get; set; } 

        public event EventHandler<ValueEventArgs<bool>> IsReadyChanged; 

        ... 
    } 

  2. The interface can be implemented onto another class that can then be registered to the Capabilities collection in ABasicDriver.

    3. Copy 
    public class SampleDriver : ABasicDriver, ITcp, IPowerBankContainer, IReadyIndicator 
    { 
        ... 

        public SampleDriver() 
        { 
            var readyIndicator = new ReadyIndicatorImplementation(); 
            Capabilities.RegisterInterface(typeof(IReadyIndicator), readyIndicator); 
        } 

        ... 
    } 

    public class ReadyIndicatorImplementation : IReadyIndicator 
    { 
        public bool IsReady { get; set; } 

        public event EventHandler<ValueEventArgs<bool>> IsReadyChanged; 
    } 


For the provided Power Controller Sample driver, the first method was used to implement capabilities since that works well for simple devices. The second method is helpful when the driver becomes large and you want to break components out into separate classes that each handle a single capability.

Implement the IPowerBankContainer Capability

The IPowerBankContainer capability is mandatory for power controllers as it is the main capability that defines your driver as a power controller to consuming applications. This capability has two primary features:

  1. The power bank collection feature
  2. The power on sequence feature (optional)

Only the power bank collection feature will be discussed in this section. The power on sequence feature is dependent on the power controller model and its implementation is straightforward.

First, create a class that implements IPowerBank. This interface defines a power bank on your power controller. The power bank may be a single outlet, a group of outlets, or a metering device such as an ammeter. The capabilities that are implemented by the power bank further define the power bank properties.

Next, implement the capabilities of the power bank within IPowerBank. These capabilities are the same as those used for the power controller. For more information on these capabilities, refer to Overview.

These capabilities can be implemented within IPowerBank similar to the two methods described in Implement Device Capabilities. The second method differs slightly in this case because there is no base class implementation (such as ABasicDriver). You must implement the TryGetCapability method where you can either return your capability class (if using the second method), or cast your power bank to the requested capability (if using the first method). You can also use a hybrid method as shown below:

Copy 
public T TryGetCapability<T>() where T : class, IDeviceCapability 
{ 
    if (typeof(T) == typeof(IPowerBankControls)) 
        return _powerBankControls; 

    return this as T; 
}

Once you implement the power bank class, you must create one instance per power bank on your power controller. Those instances must be returned in an array from the GetPowerBanks method. If there is a dynamic number of power banks and can change, the PowerBanksCollectionChanged event must be raised.

Implement the Protocol Class

The protocol class is used to handle communication to and from the device. This class will include commands that are sent from the device and will parse data received from the device.

Create a new class that implements ABaseDriverProcotol. The ConnectionChangedEvent and ChooseDeconstructMethod abstract method must also be implemented.

  • ConnectionChangedEvent will be fired when IsConnected changes. This is determined by the IsConnected flag and the ConnectionChanged event for the transport class.
  • ChooseDeconstructMethod can be used to handle responses from the device by command group. The responses must be first validated using a ResponseValidator delegate assigned to the ValidateResponse property.

To bypass ChooseDeconstructMethod and handle responses from the device as they come in, override the DataHandler method. This is easier for devices that do not have complicated responses.

The Poll method can be overridden to send out polling commands as needed. This method will be called at an interval determined by the PollingInterval.

The following sample code shows an implementation of the protocol class. For the complete sample code, refer to the Power Controller Sample driver provided in the Crestron Drivers SDK.

Copy 
public class PowerControllerProtocol : ABaseDriverProtocol 
{ 
    public PowerControllerProtocol(ISerialTransport transport, byte id) : base(transport, id) 
    { 
    } 

    protected override void ConnectionChangedEvent(bool connection) 
    {

    }

    {

    }

    {

    { 
    } 
}

Implement the Transport Class

You can choose to use one of the transport classes that exist already in the Crestron Drivers framework, such as TcpTransport or HttpsTransport. However, these transport classes are sometimes too limited for what the device requires. In that case, you must create your own transport class.

To create a custom transport class, create a new class that implements ATransportDriver and overrides SendMethod so that you can use any transport that you need. SendMethod is called by the Protocol class that handles queuing commands.

For a full implementation of a custom transport class, refer to SampleTransport.cs in the Power Controller Sample driver provided in the Crestron Drivers SDK.

Build the Driver PKG File

Once your driver code is complete, you must build it into a PKG file:

  1. Build the driver solution.
  2. Copy the DLL file from the output to the folder containing the Manifest Utility tool. Refer to the README.txt file in this folder for additional information about this tool.
  3. Run the Manifest Utility tool by double clicking ManifestUtil.exe.

If the build is successful, a PKG file with the same name as the DLL file will be created.

NOTE: Do not change the name of the PKG file. It must match the DLL file.

To verify the contents of a PKG file, it can be unzipped using 7-Zip or similar utility.

Test the Driver

Once your power controller driver is completed, perform a self test of your driver as described in Test a Driver. Then, follow the Power Controller Test Plans and ensure your driver passes all applicable test cases before releasing the driver publicly.
