AV Switcher Drivers

This topic describes how to create a Crestron Driver for a AV switcher device. AV switchers allow audio and video signals to be routed from input sources to displays and other similar devices.

NOTE: Version 4.x of the SDK framework must be used when creating AV switcher drivers. For more information on versioning, refer to SDK Framework Versioning.

Overview

AV switchers are represented by the interface IAudioVideoSwitcher.

Copy 
namespace Crestron.RAD.Common.Interfaces
{
    /// <summary>
    /// Represents a third-party audio/video switcher device.
    /// </summary>
    public interface IAudioVideoSwitcher :
                        IAudioVideoSwitchingCapabilities,
                        IHardwareDiagnostics,
                        IPower, IPower2,
                        IBasicInformation, IBasicInformation2,
                        ICustomCommand,
                        ICustomCommandCollection, ICustomCommandCollection2
                        IBasicLogger,
                        IDisposable,
                        IConnection, IConnection2, IConnection3,
                        IAuthentication3
    {
        /// <summary>
        /// A unique identifier for this switcher.
        /// </summary>
        string AudioVideoSwitcherId { get; }
    }
}


The framework represents every input and output on the device with the interface IAudioVideoExtender. It allows an application to tell the driver to route an input to an output.

Each extender object has a variable amount of AV inputs or outputs depending on its input or output type. Therefore, one extender can represent either a direct output on the switcher, such as an HDMI® port, or it could represent the last piece of equipment connected to a sync device via an HDBaseT® connection to the switcher. In the case where the last output connected to the sync is not the switcher port, the last endpoint would be the listed extender.

Using DM® switchers as an example, a DM output card would not be listed as the output; the listed output would be the DM‑RX‑200‑C connected to it if detected.

Each extender object also implements several component interfaces to allow applications to control anything that would be specific to that input/output, such as the power state or volume level.

The framework relies on the driver developer to create a JSON file that defines that configuration of the switcher by either specifying the static configuration, or listing compatible extenders and handling the assignment of them at runtime if autodetection is possible. The majority of the functions an extender performs are routed through the protocol class to allow driver developers to insert custom logic.

Create the Driver Project

For information on creating the driver project and files for an AV switcher driver, refer to Create a Driver.

Create the JSON File

The majority of the JSON file is formatted similar to other device types. The differences and feature additions are covered here. Refer to the SDK Framework for more information about the other existing nodes.

StandardCommands

This defines all commands that the driver can send to the device, either to control it or to request feedback. Methods such as PowerOn() will build a CommandSet instance using the data from this dictionary.

AV Switcher Standard Commands

Name Description and Use
PowerOn Used to build command for IPower.PowerOn.
PowerOff Used to build command for IPower.PowerOff.
Power Used to build command for IPower.Power.
PowerPoll Used to get the current state of IPower.PowerIsOn.
AudioVideoSwitcherRoutePoll Used to get the current state of IAudioVideoExtender.VideoSourceExtenderId.
AudioVideoSwitcherRoute Used to build a command for IAudioVideoSwitchingCapabilities.RouteVideoInput() and RouteAudioInput.

By default, {input} and {output} will be replaced with the API identifiers for the specified input/output extenders.

By default, the framework does not support audio breakaway. Separate video and audio routes must be handled in the protocol class via overrides.
TemperaturePoll Used to get the current state of IHardwareDiagnostics.CurrentTemperature.
TemperatureUnitPoll Used to get the current units of measurement for temperature feedback.


An example of these standard commands are provided in the following sample code:

Copy 
{
    "StandardCommands": {
        "PowerOn": {
            "Command": "POWER_ON"
        },
        "PowerOff": {
            "Command": "POWER_OFF"
        },
        "Power": {
            "Command": "POWER_TOG"
        },
        "PowerPoll": {
            "Command": "POWER__POLL"
        },
        "AudioVideoSwitcherRoutePoll": {
            "Command": "ROUTE?"
        },
        "AudioVideoSwitcherRoute": {
            "Command": "ROUTE_{input]_TO_{output}"
        },
        "TemperaturePoll": {
            "Command": "TEMP?"
        },
        "TemperatureUnitsPoll": {
            "Command": "TEMPUNITS?"
        },
    }
}


The value for “Command” is what is sent to the device. Driver developers can put tags in here that can be replaced at runtime within a method in the protocol class named PrepareStringThenSend.

NOTE: “{input}” and “{output}” are tags that are handled by the base logic.

Feedback

Feedback specifies all the information the framework needs to process feedback from the device. This follows the same pattern as other device types, but it adds the ability to specify a common delimiter that the framework can process and remove before calling any DeConstruct<Feature> methods in the protocol class.

Copy 
{
    "Feedback": {
        "Header": "->",
        "ApiDelimiter": "\r",
        "CommandAckNak": {
            "Ack": "OK",
            "Nack": "NOPE"
        },
        "PowerFeedback": {
            "Feedback": {
                "On": "ON",
                "Off": "OFF",
            },
            "GroupHeader": "POW="
        },
        "TemperatureFeedback": {
            "ValueGroupHeader": "TEMP=",
            "UnitGroupHeader": "UNITS=",
            "MetricIdentifier" "METRIC",
            "StandardIdentifier": "STANDARD"
        },
        "RoutedExtenderFeedback": {
            "GroupHeader": "ROUTED="
        }
    }
}


The driver developer may have to override ValidateResponse in the protocol class if the API does not fit this model exactly. It is recommended to make the response fit the model here and rely on the base ValidateResponse call to handle the response. This device type does not use a separate class for response validation.

The default logic of ValidateResponse is as follows. The format of this logic is [Header][GroupHeader][Feedback/Identifier][Delimiter].

  1. Waits for the response to start with [Header].
  2. Wait for the response to end with [Delimiter].
  3. Removes both [Header] and [Delimiter] from the response.
  4. Cycles through all group headers and attempts to match the response to one of them. If no group headers are found, logic then cycles through all headers found in each output or input.
  5. Sets ValidatedRxData.CommandGroup / SubCommandGroup to the appropriate value
  6. Calls the protocol class’s ChooseDeConstructMethod if this was a response for the switcher, which then switches on the command group to call the correct DeConstruct<Feature> method.

AudioVideoSwitcher

This node contains the configuration specific to the AV switcher. It lists the inputs/outputs and if they are fixed, such as on an HDMI switcher where the inputs and outputs will always be HDMI connectors that cannot be changed at runtime.

  • “NullInputIdentifier” is the string the framework would match when the current route response specifies no input routed to a particular output.

  • “<Feature>ScalingInfo” is data a developer can provide to let the framework handle scaling a percentage to/from the API’s value range. Step-size is used to increment or decrement the values.

Refer to the following sample code:

Copy 
{
    "AudioVideoSwitcher": {
        "AutoDetectConfiguration": false,
        "SupportsAudioBreakaway": false,
        "NullInputIdentifier": "NONE",
        "VolumeScalingInfo": {
            "MaximumDeviceLevel": 100,
            "MinimumDeviceLevel": 0,
            "DeviceStepSize": 1
        },
        BassScalingInfo": {
            "MaximumDeviceLevel": 10,
            "MinimumDeviceLevel": -10,
            "DeviceStepSize": 1
        },
        TrebleScalingInfo": {
            "MaximumDeviceLevel": 10,
            "MinimumDeviceLevel": -10,
            "DeviceStepSize": 1
        },
    }
}


Routable inputs and outputs are specified here. The framework refers to these components as “AudioVideoExtender” and “Extender”. Each extender is able to define their own standard commands and feedback handling. AudioVideoExtender implements its own component interfaces:

Copy 

public interface IAudioVideoSwitcher :
                    IAudioVideoSwitchingCapabilities,
                    IHardwareDiagnostics,
                    IPower, IPower2,
                    IBasicInformation, IBasicInformation2,
                    ICustomCommand,
                    ICustomCommandCollection, ICustomCommandCollection2
                    IBasicLogger,
                    IDisposable,
                    IConnection, IConnection2, IConnection3,
                    IAuthentication3


The driver developer will not have to implement IAudioVideoExtender on their own—a default implementation is provided as AudioVideoExtender that is built from the data in this node. A driver is not expected to create instances of these—they are automatically created when the driver initially loads the JSON.

Each extender will define its own set of audio and video input/outputs to represent what connectors it has available. For example, an HDMI Input card would have a one audio and video input detail with the type set to Hdmi1. The type’s index value (the “1” in “Hdmi1”) is also equal to the extender’s position in the switcher. For example, the fifth HDMI input on a switcher would have one audio and video input detail that sets type to Hdmi5. If this were an HDBaseT input, then the type would be Input5.

Extender Information (Used as Inputs or Outputs)

Name Type Description and Use
ApiIdentifier String The string used when this extender is being specified using the API.
Id String A unique identifier for this input. All input/output extenders must have unique IDs.
Name String A friendly name of this extender based on the third-party device's labels on the physical devices
Manufacturer String The extender manufacturer name. In most cases this will be the same manufacturer specified by the driver, but in some cases a third‑party extender could be used.
Model String he extender model name. This must return null if Type is a direct input/output.
Type Enum Refer to Rrestron.RAD.Common.Enums.AudioVideoExtenderType in the SDK.

Valid values are as follows:

1. DirectInput: The input is directly on the device, such as an HDMI input.

2. DirectOutput: The output is directly on the device, such as an HDMI output.

3. IndirectInput: The input is connected to the switcher, but it is not on the switcher, such as transmitter.

4. IndirectOutput: The output is connected to the switcher, but it is not on the switcher, such as a receiver.

5. IndirectInputWithSwitching: An IndirectInput that also allows its local inputs/output to be switched.

6. IndirectOutputWithSwitching: An IndirectOutput that also allows its local inputs/output to be switched.
AutoInputDetails List<Object> Contains information about an input extenders audio inputs. Null if this is an output.
AudioOutputDetials List<Object> Contains information about an output extenders audio outputs. Null if this is an input.
VideoInputDetails List<Object> Contains information about an input extenders video inputs. Null if this an output.
VideoOutputDetails List<Object> Contains information about an output extenders video outputs. Null if this is an input.
Ports List<Object> Ports that are available on this extender, such as a CEC port or COM port.
DeviceSupport Dict<Enum, Bool> Features supported by this extender. This contains all CommonFeatureSupportEnum values that are specified by the component interfaces an extender implements.
StandardCommands Dict<Enum, Object> Dictionary containing all standard commands keyed by StandardCommandEnum. This contains all the data the driver will send to the device.
Feedback Object Object that defines all supported feedback by this extender.

The JSON differs depending on the type of port exposed. Valid values for Direction are defined by Crestron.RAD.Common.Enums.PortDirection. Refer to the sample code below for each port type.

Create the Main Driver Class

If you are creating an IP driver, then the class will appear as follows. The protocol class is defined in the next CS file. The transport can be a custom transport; refer to ATransportDriver Class and Transports for more information on custom transports.

namespace AudioVideoSwitcher_Crestron_FakeDriverModel_IP
{
    using Crestron.SimplSharp;
    using Crestron.RAD.Common.Interfaces;
    using Crestron.RAD.Common.Transports;
    using Crestron.RAD.DeviceTypes.AudioVideoSwitcher;

    public class FakeAudioVideoSwitcherIp : AAudioVideoSwitcher, ITcp
    {
        public void Initialize(IPAddress ipAddress, int port)
        {
            var tcpTransport = new TcpTransport
            {
                EnableAutoReconnect = EnableAutoReconnect,
                EnableLogging = InternalEnableLogging,
                CustomLogger = InternalCustomLogger,
                EnableRxDebug = InternalEnableRxDebug,
                EnableTxDebug = InternalEnableTxDebug
            };

            tcpTransport.Initialize(ipAddress, port);
            ConnectionTransport = tcpTransport;

            AudioVideoSwitcherProtocol = 
                new FakeAudioVideoSwitcherProtocol(ConnectionTransport, Id)
            {
                EnableLogging = InternalEnableLogging,
                CustomLogger = InternalCustomLogger
            };

Make the Driver Protocol Class

The driver protocol class is where is all feedback is processed and commands are made and sent to the transport. This is where developers can override many methods to handle feedback or commands differently.

View AAudioVideoSwitcherProtocol in an object browser to see all available methods to override.

An empty protocol class that relies completely on JSON is shown in the sample code below.

Copy 
namespace AudioVideoSwitcher_Crestron_FakeDriverModel_Serial
{
    using Crestron.RAD.Common.Transports;
    using Crestron.RAD.DeviceTypes.AudioVideoSwitcher;

    public class FakeAudioVideoSwitcherProtocol : AAudioVideoSwitcherProtocol
    {
        public FakeAudioVideoSwitcherProtocol(ISerialTransport transport, byte id)
            : base(transport, id)
        { }
    }
}


By default, the framework does not handle routed feedback. If the driver will support feedback about the current routes, then it must override DeConstructSwitcherRoute. There are some methods added to AAudioVideoSwitcherProtocol that will aid developers in getting extenders.

Copy 
 protected override void DeConstructSwitcherRoute(string response)
        {
            // Receiving: ROUTED=OUTPUT#1:INPUT#1
            //            ROUTED=<output ID>:<input ID>
            //            ROUTED= is stripped out of response before this is called. 

            var routePath = response.Split(':');
            AudioVideoExtender inputExtender = null;
            AudioVideoExtender outputExtender = null;

            // We can get the extender objects here using the API identifier set
            // in the embedded file.
            // We can also get the extender objects by their unique ID using GetExtenderById
            outputExtender = GetExtenderByApiIdentifier(routePath[0]);
            inputExtender = routePath.Length > 1 ? GetExtenderByApiIdentifier(routePath[1]) : null;
            
            // Figured out which input is routed to the specified output
            // Now update the output extender with the current source routed to it
            // The framework will figure out if this was a real change or not if it is not done here.
            if (outputExtender != null)
            {
                outputExtender.VideoSourceExtenderId = inputExtender == null ? 
                    null : inputExtender.Id;
            }
        }


Some devices may send multiple messages in one burst to the transport. These can be split into individual messages by overriding DataHandler, which is the method called by the transport that will enqueue the response in an internal response queue.

Copy 
public override void DataHandler(string rx)
        {
            // Route poll responses come in a large chunk and need to be processed seperatly.
            // Add them to the response queue and keep the delimiter
            var delimiterIndex = rx.IndexOf('\r');
            if (delimiterIndex > 0 &&
                rx.IndexOf('\r', delimiterIndex + 1) > 0)
            {
                var splitRx = rx.Split('\r');
                for (int i = 0; i < splitRx.Length; i++)
                {
                    base.DataHandler(splitRx[i] + '\r');
                }
            }
            else
            {
                base.DataHandler(rx);
            }
        }


Some drivers may need to add a checksum or a delimiter to commands being sent to the device. Delimiters can be defined on each command in the JSON file, or they can all be handled with one override.

Copy 
        protected override bool PrepareStringThenSend(CommandSet commandSet)
        {
            if (!commandSet.CommandPrepared)
            {
                commandSet.Command = string.Format("{0}\x0D", commandSet.Command);
                commandSet.CommandPrepared = true;
            }
            return base.PrepareStringThenSend(commandSet);
        }


DeConstructExtender<Feature> methods can be overridden for the main AV switcher if other actions need to be performed when that feature changes states.

Copy 
    protected override void DeConstructSwitcherPower(string response)
            {
                // Device reports three states: On, Off, and Standby
             // Treat standby as Off
             if (response.Equals("standby", StringComparison.CurrentCultureIgnoreCase))
                {
                    // In JSON, PowerOn feedback is set to "On"
                    response = "On";
                    }

                base.DeConstructSwitcherPower(response);
             }


DeConstructExtender<Feature> methods for extenders can also be overwritten.

Copy 
        protected override void DeConstructExtenderClippingAudio(AudioVideoExtender extender, string response)
        {
            // Device response can be unknown while JSON only specifies on/off. 
            // Treat unknown as off
            if (response.Equals("unknown", StringComparison.CurrentCultureIgnoreCase))
            {
                response = "Off";
            }

            base.DeConstructExtenderClippingAudio(extender, response);
        }


An example of overriding a DeConstructExtender<Feature> and not calling the base is shown below. The setters for all public properties that are defined by IAudioVideoExtender will throw the appropriate events if set to a different state. This example shows what the base method does by default so this override would not be needed.

Copy 
        protected override void DeConstructExtenderPower(AudioVideoExtender extender, string response)
        {
            if (response.Equals("on", StringComparison.CurrentCultureIgnoreCase))
            {
                extender.PowerIsOn = true;
            }
            else
            {
                extender.PowerIsOn = false;
            }
        }

Add Capabilities to Inputs and Outputs

As of Driver Framework 25, IAudioVideoExtender2 has been added to allow inputs and outputs to specify supported capabilities when using IDeviceWithCapabilities. Developers can override the method AddCapabilitiesToExtenders in AAudioVideoSwitcherProtocol to add their own custom capabilities.

NOTE: Only IVideoSyncDetection is supported at this time.

Refer to the following example capability implementation.

Copy 
public class VideoSyncDetectionCapability : IVideoSyncDetection
    {
        private bool _videoSyncDetected;
        public bool VideoSyncDetected
        {
            get { return _videoSyncDetected;}
            set 
            {
                if (_videoSyncDetected != value)
                {
                    _videoSyncDetected = value;

                    var videoSyncDetectedChanged = VideoSyncDetectedChanged;
                    if (videoSyncDetectedChanged != null)
                    {
                        videoSyncDetectedChanged(this, EventArgs.Empty);
                    }
                }
            }
        }
        public event EventHandler VideoSyncDetectedChanged;
    }

The following example shows capabilities being added to AudioVideoExtender in AAudioVideoSwitcherProtocol.

Copy 
protected override void AddCapabilitiesToExtenders(IEnumerable<AudioVideoExtender> inputs, IEnumerable<AudioVideoExtender> outputs)
    {
        foreach (AudioVideoExtender extender in inputs)
        {
            AddCapabilityToExtender(extender, typeof(IVideoSyncDetection), new VideoSyncDetectionCapability());
        }
    }
