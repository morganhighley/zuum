Crestron Home Sequences

When the Sequences menu is selected in the Crestron Home UI, navigating into a room will show all available devices and entities that are available to be programmed within the room. Crestron Home devices can now be added as programmable devices.

Crestron Home devices are sorted by device type/name and, when selected, will have operations associated with them that can be added as a step to a sequence.

Below is a screen shot of a Crestron Home device in a room and its operations.

Sample Crestron Home Device in a Room

The name associated with the Crestron Home device is the same name used while adding the device in the Pair Devices page. The icon used will be dependent on the Crestron Home device type.

Selecting the device will show all operations that are available for programming. Operations available for programming will be controlled via declarative attributes in the Crestron Home device driver code.

Attributes

The following attributes are used to control the Crestron Home device operations that are available for programming.

ProgrammableOperationAttribute

To get a method or property for a Crestron Home device to appear as an operation in sequences automatically, the method or property should be decorated with the ProgrammableOperationAttribute. Refer to the following sample code.

Copy 
public class DoorLock
{
    [ProgrammableOperation]
    public void LockDoor()
    {
    
    
    }
    
    [ProgrammableOperation]
    public int SetDoorId { get; set; }
}

This attribute can be applied to methods or properties regardless of scope (public, private, internal, and so forth).

The name of the method or property will be used by default as the operation name in sequences. In the example below, the “LockDoor” and “SetDoorId” operation names in sequences match the method names in the Crestron Home driver code above.

Operation Names for Sequences

To control the operation name, the display name parameter can be used. The code below shows the operation name in sequences as “Lock Door” (refer to Localization for more information on localizing operation names) even though the method name “LockInteriorDoor” was used.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockInteriorDoor()
{


}

This name is reflected in the Sequences menu.

Sequence Menu - Display Name Parameter

Methods that take parameters can also be decorated with the ProgrammableOperationAttribute. By default, a parameter dialog will appear in sequences automatically to allow the user to input the parameters specified. The UI controls used for this input are dependent on the data type of each parameter (text box for strings, checkbox for Booleans, and so forth).

NOTE: At this time, only primitive (int, bool, and so forth) data types are supported.

The code below shows a method taking a parameter and the parameter dialog that will be displayed automatically when this operation is added to a sequence.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor(int time)
{


}

This information is reflected in the image shown below.

Method Taking Parameter UI Example

By default, the label used for a parameter is the parameter name specified in the method. To control the parameter name, use the DisplayAttribute.

Once the operation is added to a sequence, parameters entered will be shown in the sequence step automatically.

Parameters Added to Sequence Step UI Example

DisplayAttriute

The DisplayAttribute is used to control the text of the label used while gathering user input for a parameter. Refer to the sample code below.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor ([Display("Seconds")] int time)
{


}

The following image shows that the label in the UI is updated to "Seconds" as indicated by the DisplayAttribute.

DisplayAttribute UI Example

UnitAttribute

The UnitAttribute is used to display a unit next to the parameter label. Multiple attributes can be applied to a single parameter. Refer to the sample code below.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    int time)
{


}

The following image shows that the seconds unit ("s") is displayed next to the label as indicated by the UnitAttribute.

UnitAttribute UI Example

The following units are supported:

Copy 
public enum Unit
{
    None,
    Celsius,
    Fahrenheit,
    Percentage,
    Millivolts,
    PartsPerMillion,
    GramsPerLitter,
    pH,
    Hours,
    Minutes,
    Seconds,
    Milliseconds,
    Years,
    Months,
    Days
}

DefaultAttribute

The DefaultAttribute is used to control the default value of a parameter when the parameter dialog is displayed. The data types int, double, string, and bool can be entered as a default value. Refer to the sample code below.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5)]
    int time)
{


}

[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5.1)]
    double time)
{


}

[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5)]
    string time)
{


}

[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Allow to Unlock")]
    [Unit(Unit.Seconds)]
    [Default(true)]
    bool allowToUnlock)
{


}

The following image shows the default value of a parameter as indicated by the DefaultAttribute when the parameter dialog is displayed.

DefaultAttribute UI Example

The DefaultAttribute also allows a developer to pass in true or false to a readOnly parameter, which indicates whether the parameter dialog should display UI controls for the given parameter.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5, true)]
    int time)
{


}

If true is passed into the readOnly parameter, then UI controls will not be provided for the given parameter. If the method does not have any more parameters that are not marked as read only, the parameter dialog for the operation will not be displayed at all for the given operation.

The read-only value (with the value specified in the attribute) will be used during execution of the operation.

MinAttribute

The MinAttribute should be used to set a minimum value for a parameter. Data types of type int and double can be entered as a minimum value. Refer to the following sample code.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5)]
    [Min(1)]
    int time)
{


}

[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5.1)]
    [Min(1.1)]
    double time)
{


}

If a value is entered in the parameter dialog for a parameter that is less than the minimum value specified, a validation error will occur automatically in sequences as shown in the image below.

MaxAttribute

The MaxAttribute is used to set a maximum value for a parameter. Data types of type int and double can be entered as a maximum value. Refer to the sample code below.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5)]
    [Min(1)]
    [Max(10)]
    int time)
{
    
    
}

[ProgrammableOperation("Lock Door")]
public void LockDoor (
    [Display("Time")]
    [Unit(Unit.Seconds)]
    [Default(5.1)]
    [Min(1.1)]
    [Max(10.1)]
    double time)
{


}

If a value is entered in the parameter dialog for a parameter that is higher than the maximum value specified, a validation error will occur automatically in sequences as shown in the image below.

MaxAttribute UI Example - Validation Error

Select the alert icon to receive details about the alert.

MaxAttribute UI Example - Alert Message

AvailableValuesAttribute

The AvailableValuesAttribute should be used when a list of well‑defined values is known at compile time for a parameter. Values of type int, double, and string can be entered at once. Refer to the sample code below.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoor ([AvailableValues(1, 2, 3)] int id)
{


}

[ProgrammableOperation("Lock Door")]
public void LockDoor ([AvailableValues(1.1, 2.1, 3.1)] double id)
{


}

[ProgrammableOperation("Lock Door")]
public void LockDoor ([AvailableValues("1", "2", "3", "4")] string id)
{


}

If there are no other parameters in the method specified, the operation will be displayed with a folder icon indicating that there are child elements for the operation.

DynamicAvailableValuesAttribute

The DynamicAvailableValuesAttribute is used when a list of well‑defined values is needed for a parameter, but it is not known at compile time.

At runtime, when the operation is selected, a method or property on the Crestron Home device associated with the operation will be called for the parameter’s available values. The typeName string passed in should be the type of the object that will be called and the operationName passed in should be the method or property name on the object to call. The return type of the method or property should be an IEnumerable and match the parameter’s data type.

Refer to the sample code below.

Copy 
public list<int> LockIds {get; }

[ProgrammableOperation("Lock Door")]
public void LockDoor([DynamicAvailableValues(nameof(DoorLock), nameof(LockIds), OperationType.Property)] int id)
{


}

public list<int> GetLockIds()
{
    // Business logic to return lock ids goes here.
    return new List<int>();
}


[ProgrammableOperation("Lock Door")]
public void LockDoorWithId([DynamicAvailableValues(nameof(DoorLock), nameof(GetLockIds), OperationType.Method)] int id)
{


}

The type of object and operation to call does not have to be at the root object. An optional path from the root object using dot notation n levels deep can also be specified.

In the example below, the current object has a Device property and the object returned has a Registry property. The Registry object is of type ILockRegistry, so the GetLockIds method will be called on the ILockRegistry object for available values.

Copy 
[ProgrammableOperation("Lock Door")]
public void LockDoorWithId (
    [DynamicAvailableValues(
        nameof(ILockRegistry),
        nameof(GetLockIds),
        OperationType.Method,
        "Device.Registry")]
    int id)
{


}

Once the available values are retrieved, the same workflow described in AvailableValuesAttribute applies here.

IsSupportedAttribute

The IsSupportedAttribute is used when an operation in sequences should only exist under certain business conditions.

For example, if the same type of door lock device is used to represent different kinds of door lock models, and on certain types of door lock models the unlock operation is not supported, the unlock operation should be decorated with the IsSupportedAttribute.

Refer to the sample code below.

Copy 
public bool GetIsUnlockSupported()
{
    // Business logic on whether or not unlock is supported goes here.
    return true;
}


[ProgrammableOperation]
[IsSupported(nameof(DoorLock), nameof(GetIsUnlockSupported), OperationType.Method)]
public void Unlock()
{


}

The same semantics of typeName, operationName, and an optional path property apply here as they did in DynamicAvailableValuesAttribute.

Before the operation is displayed in sequences, the method or property specified in the IsSupportedAttribute will be called. If the operation is not supported (the method or property called returns false), the operation will not be displayed in sequences.

Localization

All strings in the attributes above can be localized by prepending the string with a “^” symbol. This is consistent with other strings that are localized in Crestron Home devices (UiDefinition.xml). Refer to the sample code below.

Copy 
[ProgrammableOperation("^Lock Door")]
public void LockDoor (
    [Display("^Seconds")]
    int time)
{


}

When sequences encounter a string that needs to be localized, the Crestron Home device will be queried for the translation string at runtime, and the localized string will be displayed for the applicable item. If the localized string is not found, the string will be displayed as is with the “^” symbol as shown in the image below.

Localized String Not Found UI Example

Package Generation

The above attributes and programming will be included in the PKG file when the Manifest Utility is ran against the compiled driver’s DLL file. Be sure to pass in the full path of the DLL when running ManifestUtil.
