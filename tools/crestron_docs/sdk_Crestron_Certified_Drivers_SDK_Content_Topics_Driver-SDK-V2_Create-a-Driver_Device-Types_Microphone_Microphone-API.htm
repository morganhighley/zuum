Microphone API Reference

This section lists the various properties, commands, events, and types that make up the Device Driver Entity Model API for Microphone devices.

Common

commandFunction

Defines an enumeration to be used for press and hold functionality across many driver capabilities

Types

crestron:CommandFunction

Fully Qualified Id: crestron:CommandFunction

Describes the operation of a command which may be issued once or may be repeated for some duration.

The most common use case for this is remote control button emulation. A single tap would send a pulse, but pressing and holding the button would send high until the button is released and then low at the end.

Sending high and then low immediately is not equivalent to sending pulse in all cases. Applications should make their own determination on when to treat a press as a pulse versus high/low.

Definition
Copy 
[EntityDataType(Id = "crestron:CommandFunction")]
    public enum CommandFunction
    {
        /// <summary>
        /// Indicates a single, momentary operation.
        /// If <see cref="High"/> was previously used, this stops that operation and begins the
        /// new single pulse operation.
        /// </summary>
        Pulse,
    
        /// <summary>
        /// Indicates the beginning of an ongoing operation.
        /// If <see cref="High"/> was previously used, this will have no effect since it is
        /// already started.
        /// Callers MUST ensure they follow up with <see cref="Low"/> eventually to avoid a
        /// runaway condition where the operation goes on forever.
        /// </summary>
        High,
    
        /// <summary>
        /// Indicates the end of an ongoing operation.
        /// If <see cref="High"/> was not previously used, this will have no effect since it is
        /// already stopped.
        /// </summary>
        Low,
    }

Defines the crestron:DeviceUxCategory enumeration

Types

crestron:DeviceUxCategory

Fully Qualified Id: crestron:DeviceUxCategory

The User Experience category for a device

This describes the type of device that it is. It should ONLY be used for categorization purposes for selecting an icon or helping organize the device into a list. It should NOT be used to determine how to use the device.

In practice, devices can fit into multiple of these categories. The driver developer may have to choose what they consider the closest, or use "other" if no other categories apply. Applications should inspect the definition of the device instance to determine its capabilities and assume nothing about the device based on this string.

deviceName

Capability to be implemented by a device or device component which supports a configurable name

This capability is reserved for devices which report their own user-configurable name, and potentially the ability to set that name via the API.

This capability should not be "faked" where the driver emulates this feature.

This "name" is for describing and identifying the device in some user interface. Sometimes the device supports configuring its host name on the network, but that is not the intent of this interface.

This is NOT the label associated with a device component. For that, see the deviceComponentLabel capability.

For example, a power controller may have multiple banks, labeled something like "Bank 2" or "Aux" on the power controller chassis. It may also support a software-defined "name" for the bank that may be changed via the device user interface. This capability would be used to report that software-defined name, not the "Bank 2" or "Aux" label.

audioMixer

This capability exposes collections of audio nodes (channels) that are inputs, outputs, or internal to an audio mixer device.

There are essentially 3 types of "channels":

Input channels: channels that are inputs into an audio mixer device. The physical connections may be wireless and/or wired Examples include microphones, media streaming services, and AVR/display audio output connections.

Mix channels: channels that have 1 or more input and output sources (and a gain for each). The sources may includes input channels or other mix channels. The outputs may feed into other mix channels or output channels.

Output channels: channels that are outputs of an audio mixer device. The physical connections may be wireless and/or wired A single audio mixer may have more than one output channel. Examples include speakers, output streams, and output connectors that may be fed externally to an AVR audio input, amplifier, or even another audio mixer.

mediaContentContext

Provides hints to the application as to how a media source or destination (or input or output, although these are less common) is intended to be used by its manufacturer.

DO NOT USE THIS TO DETERMINE "DEVICE TYPE"!

This ONLY provides hints for setting up reasonable default configurations. A device may still span the lines and qualify as multiple devices. Another device might not implement these at all but be perfectly suitable to use. ALWAYS use the capabilities of the device to determine what it supports and what use cases it qualifies in your system. ONLY use this to provide automatic default behavior that is already manually configurable by the user if this capability did not exist.

For example, an application that handles a conference room should allow for any audio source to be treated as a room microphone since there are a number of ways a microphone signal could be wired into the system. However, if the user adds a dedicated microphone device to the system, the application may desire to add that as a room microphone by default (even though it is still configurable beyond that) as a convenience.

Similarly, if a room contains both a display with integrated speakers and an audio amplifier, the application may wish to default to using the display for video and the amplifier for audio (again, only as a default setup for convenience and not as an enforced rule).

Types

Suitability

Fully Qualified Id: mediaContentContext:Suitability

Describes tiers of suitability for a given use case

A value of null (or equivalent default value in the programming environment) or unimplemented property is equivalent to a suitability of "unknown"

Levels are thoroughly defined so that an application may compare them and have less of a chance of encountering a future-defined level and not knowing where to order it. In most cases, this level of granularity is not needed. Devices should use the preferred values as indicated in the description unless they have a very specific necessary reason to use a reserved one.

Note that an application may not look at these levels at all, or it may not care to distinguish between all of these levels. They are provided only as additional information the application could choose to use to enhance the user experience.

devicePower

This capability describes the overall power state of a device which needs to be powered on before it can be controlled

This is intended to control power on a device but not start its operation. For example, it can be used to turn on a washing machine so that it can later be started in a specific separate API call.

This does NOT apply to actions like turning a light on or powering a bank on a power controller or turning on a fan. Those actions should use their own specific API. This capability is solely intended to be used to power on a device so that it may be controlled, not to cause the device to perform an action.

When a device or entity implements this capability, the application must use it to power on that device before attempting to control it.

While not limited to media devices, this capability is commonly implemented on them. A media device with no permanently active outputs fits the above descriptions perfectly. However, it is worth noting that many devices do have permanently active outputs, and powering on those outputs is likely to start that device's primary function (such as turning on a display). However, in the context of media, a display is a controllable media destination, not a separate "device" per-se. The API only cares that the media destination is powered on, not that it is a display specifically. That, in addition to the fact that displays do not typically support discrete power control over their internal processor and the display screen itself and thus a separate power property and command would be redundant, is the justification for media devices appearing to "break the rule" for the use of devicePower.

Sets the color of the indicator using an RGB value

Since intensity may conflict with colorRgb, a driver shall not implement both setIntensity and setColorRgb.

Parameters
Name Details
rgb

The rgb color setting to set the indicator to. Should be a string in RRGGBB format (where RR, GG and BB are hex digits).
