Driver SDK V1 (RAD Framework) / Create a Driver / Nonbasic Device Types / Overview

Overview

A platform driver permits communication with one or more paired devices through a gateway. The paired devices and the gateway together make a platform.

NOTE: The term “paired device” is used to refer to any device that is paired with or is a child of the gateway. The term “gateway” is used to refer to the device or service that is communicating to control the paired device(s). Depending on the actual device, the term gateway may refer to a physical gateway (such as a lighting hub), or it may refer to a cloud‑based service that returns a list of all the devices associated to a specific user’s account.

A platform driver should be used over a standard driver when the driver will contain one or more devices that communicate through the same API. An example of this is a third‑party lighting hub, where the hub serves as the “gateway” to communicate with the individual bulbs. In this scenario, a platform driver can be configured where the lighting hub is the gateway and the individual bulbs are the paired devices. The platform driver allows the integrator to add the gateway to the system, which will then tell the consuming application which devices are paired. The end result is that each bulb does not need to be added as an individual driver, eliminating the need to repeatedly add the same configuration information for each bulb.

Platform drivers can have almost any type of driver as a paired device (except for media devices as described in the note below). This includes supported driver types in the framework as well as Crestron Home Extension device drivers.

NOTE: There is currently a known issue in Crestron Home where media devices do not work properly as paired devices. This issue involves the following device types: AV Receiver, AV Switcher, Bluray Player, Cable Box, Flat Panel Display, Projector, and Video Server.

Platform Driver Framework

A platform driver is made up of a single gateway class that derives from the abstract base class AGateway, which is referred to also as the “gateway class”. The gateway class contains a protocol class. The protocol class derives from AGatewayProtocol and handles all communication (to the paired devices, hub, web API, an so forth), as well as adding or removing paired devices.

NOTE: The consuming application creates an instance of the gateway class. It is the responsibility of the gateway protocol to provide the instances of the paired devices using the method AddPairedDevice in the base gateway protocol class.

AGateway Class

The following table describes the methods that are available to the consuming application. These methods are already implemented in the base AGateway class and are listed here for reference only.

Method Description and Use
GetDevices() This method is called by the consuming application to get the list of paired devices on the gateway. Adding a paired device to the gateway is done in the gateway’s protocol class.
DeviceListChanged This event is used to notify the consuming application when there is a change in the list of paired devices (for example, when a paired device is added or removed).
GetDeviceInstance<T>(string deviceId)

This method is called by the consuming application to get the instance of a paired device from the gateway.

The deviceId passed in matches the ID of the device specified by the GetDevices() method.

AGatewayProtocol Class

The following table describes the available methods for this class. These methods are used to manage the gateway’s paired devices.

Method Description and Use
AddPairedDevice(GatewayPairedDeviceInformation pairedDeviceInfo, ABasicDriver instance) This method is called by the driver protocol class to add a paired device to the gateway. pairedDeviceInfo is information about the device, which serves a similar purpose to the JSON data file of a normal driver. The instance is the actual instance of the underlying driver for the paired device.

This will raise the DeviceListChanged event.
RemovePairedDevice(string pairedDeviceId) This method is used to remove an instance of a paired device from the gateway. The pairedDeviceId should match the ID of the device to be removed, which is defined in GatewayPairedDeviceInformation class for the device.

This will raise the DeviceListChanged event.
RemoveAllPairedDevices()

This method is similar to RemovePairedDevice, except this method will remove all devices that have been paired with the gateway.

This will raise the DeviceListChanged event.

UpdatePairedDevice(string pairedDeviceId, GatewayPairedDeviceInformation newPairedDeviceInformation) This method is used to update the information of an already existing paired device that has been added using the AddPairedDevice method.

This will raise the DeviceListChanged event.

GatewayPairedDeviceInformation Class

The following table describes the properties of this class. This class is used to provide information about the paired device when adding it to the gateway, which serves a similar purpose to the JSON data file of a standard driver.

Property Description and Use
Id A unique identifier for the device that should persist across restarts. This can be a MAC address or any other device specific identifier.
Name A user-friendly name for the device. Ideally, this should be a name assigned to the device by the user.
Description A user-friendly description of the device.
Manufacturer The device manufacturer name.
Model The device model name.
DeviceType The device type, which should match a value from the Crestron.RAD.Common.Enums.DeviceTypes enum.

If the device type is not listed in the DeviceTypes enumeration, a custom value may be used, but only if the device of the Crestron Home Extension type (the paired device instance must inherit AExtensionDevice).
DeviceSubtype The device subtype. This field is optional and can be used to further specify the type of the device (for example, DeviceType = Appliance, DeviceSubtype = Dishwasher).

Platform Driver PKG File

A platform driver is in the form of a PKG file like all other drivers. However, platform driver PKG files have certain unique elements:

  • The PKG file contains the DLL file for the platform driver as well as a DLL file for each of the paired device drivers.
  • If the platform driver contains Crestron Home Extension devices as paired devices, then the PKG will also contain a single uidefinitions folder and translations folder inside.
    • Each paired Crestron Home Extension device will share the same uidefinitions folder, so each Crestron Home Extension driver must have to be a unique UI definition. The Crestron Home Extension driver will then have to override the GetUiDefinition method and select the correct UI definition on its own.
    • The same applies for translations: each translation file will need to contain the translations for all Crestron Home Extension drivers in the platform driver.
