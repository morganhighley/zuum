Crestron Home Programming

Defining custom commands, properties, and events for use in sequences is done the same way as standard ones defined by Crestron, such as light:isOn.

NOTE: Custom items should not have a namespace:. For example, use lightIsOn instead of light:isOn. This ensures no conflict with any future‑defined standard items.

When the Sequences menu is selected in the Crestron Home UI, navigating into a room will show all available devices and entities that are available to be programmed within the room. Crestron Home devices can now be added as programmable devices. Any commands, events, and properties tagged as Programmable will be displayed in a room other devices. They are sorted by device type/name and, when selected, will have operations associated with them that can be added as a step to a sequence.

Below is a screen shot of a Crestron Home device in a room and its operations.

Sample Crestron Home Device in a Room

The name associated with the Crestron Home device is the same name used while adding the device in the Pair Devices page. The icon used will be dependent on the device type in the driver’s JSON file, or the crestron:uxCategory of the device if it is a child of a platform.

Selecting the device will show all operations that are available for programming. Operations available for programming will be controlled via declarative metadata in the device’s reported DriverControllerDefinition.

The sections that follow describe the commands, properties, and events that can be used for Crestron Home programming.

Commands

The following command attributes are used to control the Crestron Home device operations that are available for programming. It is also possible to declare these operations programmatically by providing the appropriate metadata when constructing the command.

NOTE: As a best practice, driver commands should be made available to Crestron Home programming even if the device is offline with the control system.

EntityCommandMetadataAttribute

To get a device command to appear as an operation in sequences automatically, the method should be decorated with EntityCommandMetadataAttribute(Programmable = true). Refer to the following sample code.

Copy 
public class DoorLock 
{ 
    [EntityCommand] 
    [EntityCommandMetadata(Programmable = true)] 
    public void LockDoor() 
    { 
     
     
    } 
} 

This attribute can be applied to methods regardless of scope (public, private, internal, and so forth).

The name of the method will be used by default as the operation name in sequences.

To control the operation name, the FriendlyName parameter can be used. The code below shows the operation name in sequences as "Lock Door" even though the method name “lockInteriorDoor” was used.

Copy 
[EntityCommand(FriendlyName = "Lock Door")] 
[EntityCommandMetadata(Programmable = true)] 
public void LockInteriorDoor() 
{ 
 
 
} 

This name is reflected in the Sequences menu.

Sequence Menu - FriendlyName Parameter

Methods that take parameters can also be marked programmable. By default, a parameter dialog will appear in sequences automatically to allow the user to input the parameters specified. The UI controls used for this input are dependent on the data type of each parameter (text box for strings, checkbox for Booleans, and so forth).

NOTE: At this time, only primitive (int, bool, and so forth) data types are supported.

The code below shows a method taking a parameter and the parameter dialog that will be displayed automatically when this operation is added to a sequence.

Copy 
[EntityCommand(FriendlyName = "Lock Door")]
[EntityCommandMetadata(Programmable = true)]

public void LockDoor(int time)
{


}

This information is reflected in the image shown below.

Command Taking Parameter UI Example

By default, the label used for a parameter is the parameter name specified in the method. To control the parameter name, use the FriendlyName parameter of EntityParameterAttribute.

Once the operation is added to a sequence, parameters entered will be shown in the sequence step automatically.

Parameters Added to Sequence Step UI Example

FriendlyName

The FriendlyName parameter of EntityParameterAttribute is used to control the text of the label used while gathering user input for a parameter. Both the command name and the parameter name can be specified via a translation key independently of one another. Refer to the sample code below.

Copy 
[EntityCommand(FriendlyName = "Lock Door")]
[EntityCommandMetadata(Programmable = true)]
public void LockDoor( 
    [EntityParameter( 
        FriendlyName = "Seconds" 
    )] 
    int time 
) 
{ 

} 


[EntityCommand(FriendlyName = "Lock Door 2", NameLocalizationKey = "lockDoor2")]
[EntityCommandMetadata(Programmable = true)]
public void LockDoor2( 
    [EntityParameter( 
        FriendlyName = "Seconds", 
        NameLocalizationKey = "lockDoor2_time" 
    )] 
    int time 
) 
{ 

} 

The following image shows that the label in the UI is updated to "Seconds" as indicated by the FriendlyName.

FriendlyName UI Example

Units

The Units value is used to display a unit next to the parameter label. The following unit strings are preferred values defined by Crestron. Use these specific strings if they apply. For example, use “Milliseconds” and not "ms".

  • Celsius
  • Fahrenheit
  • Percentage
  • Millivolts
  • PartsPerMillion
  • GramsPerLiter
  • pH
  • Hours
  • Minutes
  • Seconds
  • Milliseconds
  • Years
  • Months
  • Days

Refer to the sample code below.

Copy 
[EntityCommand(FriendlyName = "Lock Door")]
[EntityCommandMetadata(Programmable = true)]
public void LockDoor( 
    [EntityParameter( 
        FriendlyName = "Time", 
        Units = "Seconds" 
    )] 
    int time 
) 
{ 

} 

The following image shows that the seconds unit ("s") is displayed next to the label as indicated by Units.

Parameter Units UI Example

DefaultValue

The DefaultValue is used to control the default value of a parameter when the parameter dialog is displayed. The data types int, double, string, and bool can be entered as a default value. Refer to the sample code below.

Copy 
[EntityCommand(FriendlyName = "Lock Door")]
[EntityCommandMetadata(Programmable = true)]
public void LockDoor( 
    [EntityParameter( 
        FriendlyName = "Time", 
        Units = "Seconds", 
        DefaultValue = 5 
    )] 
    int time 
)
{ 

}

The following image shows the default value of a parameter as indicated by DefaultValue when the parameter dialog is displayed.

RangeMinimum

RangeMinimum should be used to set a minimum value for a parameter. Data types of type int and double can be entered as a minimum value. Refer to the following sample code.

Copy 
[EntityCommand(FriendlyName = "Lock Door")]
[EntityCommandMetadata(Programmable = true)]
public void LockDoor( 
    [EntityParameter( 
        FriendlyName = "Time", 
        Units = "Seconds", 
        DefaultValue = 5, 
        RangeMinimum = 1 
    )] 
    int time 
)
{ 

} 

If a value is entered in the parameter dialog for a parameter that is less than the minimum value specified, a validation error will occur automatically in sequences as shown in the image below.

AvailableValues

AvailableValues should be used when a list of well‑defined values is known at compile time for a parameter. Values are entered as strings, even when the parameter is an integer or a double. The friendly name and localization key for the values may also optionally be specified. If present, that will be used in place of the raw value in the UI. If these values are specified, the number of values must match the number of names and/or localization keys.

Copy 
[EntityCommand(FriendlyName = "Lock Door")]
[EntityCommandMetadata(Programmable = true)]
public void LockDoor( 
    [EntityParameter( 
        FriendlyName = "Time", 
        Units = "Seconds", 
        AvailableValues = new [] { "1", "2", "3" } 
    )] 
    int id 
) 
{ 

} 

[EntityCommand(FriendlyName = "Lock Another Door")]
[EntityCommandMetadata(Programmable = true)]
public void LockAnotherDoor( 
    [EntityParameter( 
        FriendlyName = "Time", 
        Units = "Seconds", 
        AvailableValues = new [] { "1", "2", "3" }, 
        AvailableValuesLabels = new [] { "One", "Two", "Three" },
        AvailableValuesLocalizationKeys = new [] { "LockId_1", "LockId_2", "LockId_3" } 
    )] 
    int id 
) 
{ 

}

If there are no other parameters in the method specified, the operation will be displayed with a folder icon indicating that there are child elements for the operation.

AvailableValuesProperty

The AvailableValuesProperty is used when a list of well‑defined values is needed for a parameter, but it is not known at compile time.

At runtime, when the operation is selected, a device property value will be used for the parameter’s available values. That parameter must be an array of available values.parameter’s data type.

Refer to the sample code below.

Copy 
[EntityCommand(FriendlyName = "Lock Door")] 
[EntityCommandMetadata(Programmable = true)]  
public void LockDoor(  
    [EntityParameter(  
        FriendlyName = "Time",  
        Units = "Seconds",  
        AvailableValuesProperty = "lockDoorIds"  
    )]  
    int id  
)  
{  
 
}  
 
[EntityProperty(Id = "lockDoorIds")]  

[EntityCommandMetadata(Programmable = true)]  
public DriverEntityAvailableValue[] LockDoorIds  
{  
    get  
    {  
        return new[]  
        {  
            new DriverEntityAvailableValue("1", new DriverEntityLocalizedString("One", "LockId_1"), false),  
            new DriverEntityAvailableValue("2", new DriverEntityLocalizedString("Two", "LockId_2"), false),  
            new DriverEntityAvailableValue("3", new DriverEntityLocalizedString("Three", "LockId_3"), false),  
        };  
    }  
}  

Once the available values are retrieved, the same workflow described in AvailableValues applies here.

Properties

The following property attribute is used to control the Crestron Home device operations that are available for programming. It is also possible to declare these operations programmatically by providing the appropriate metadata when constructing the property.

NOTE: Properties are compatible with Crestron Home version 4.03 (and higher) only.

EntityPropertyMetadataAttribute

To get a device property to appear automatically as a change event in sequences and as a value in conditionals, the property should be decorated with EntityPropertyMetadataAttribute(Programmable = true) in addition to the standard EntityPropertyAttribute definition.

NOTE: The EntityPropertyAttribute contains many of the same properties as EntityParameterAttribute, which will work the same as with parameters.

Refer to the sample code below.

Copy 
[EntityProperty(FriendlyName = "Percent Property", Units = "Percentage", 
    RangeMinimum = 1,
    RangeMaximum = 100)]
[EntityPropertyMetadata(Programmable = true)]
public int PercentProperty
{
    get => _percentProperty;
    set
    {
        if (_percentProperty == value)
            return;

        _percentProperty = value;
        NotifyPropertyChanged("percentProperty", new DriverEntityValue(_percentProperty));
    }
} 

The property will now be available in conditions.

Example Property in Conditions

Additionally, an event is created when this value changes, which can be used to trigger other sequences.

Actions & Events Menu (With Property Changing Event)

The EntityPropertyMetadataAttribute can be applied to properties regardless of scope (public, private, internal, and so forth). The name of the property will be used by default as the value name in sequences if FriendlyName is not provided.

NOTE: At this time, only primitive (bool, string, double, and integer types except ulong) data types are supported.

Events

All Crestron Home driver devices with events are displayed on the Actions & Events page for the room alongside other types of programmable devices and are ordered alphabetically by device name. The manufacturer name of the device is also displayed.

If a Crestron Home device does not have any events, it will not appear in this list. Selecting the device will display all available events that can be programmed.

The Sequence mode is available for programming use.

NOTE: A future release of Crestron Home will provide the ability to program other modes such as Scene and Quick Actions.

Once the Sequence mode is chosen, a user experience is provided similar to other areas in the application.

If a device is deleted or the room in which the device is deleted, all programming associated with the device’s events will be deleted.

Programming Metadata

The following programming metadata is provided for Crestron Home device events.

EntityEventAttribute

To have an event for a device appear on the Actions & Events page automatically, the event should set the Programmable flag on the event metadata via EntityEventAttribute. Friendly Name and Localization Key, while encouraged, are both optional. If not specified, the name of the event will be used.

Copy 
[EntityEvent(Id = "locked", FriendlyName = "Door Locked", NameLocalizationKey = "Event_Locked")] 
[EntityEventMetadata(Programmable = true)] 
public event EventHandler Locked; 

[EntityEvent(Id = "unlocked")] 
[EntityEventMetadata(Programmable = true)] 
public event EventHandler Unlocked; 

TriggeredBy

Sequences that are triggered from events increase the chances of creating loops in the system. For example, the Locked event can be programmed to unlock a door lock and then the Unlocked event can be programmed to lock the same door lock.

To help reduce the risk of loops, warning messages will be displayed as applicable when operations are added to a sequence that target the same device as the event being programmed. These warning messages will be driven by the operation names specified in the TriggeredBy list.

Refer to the following sample code.

Copy 
[EntityEvent(Id = "locked", FriendlyName = "Locked")] 
[EntityEventMetadata(Programmable = true, TriggeredByCommands = new [] { "lock", "lockWithTimeout" })]
public event EventHandler Locked; 


In the code above, the TriggeredBy array is specifying that the locked event is triggered by the lock and lockWithTimeout operations. If the lock or lockWithTimeout operations are added to the Locked event sequence, the following warning message will be displayed.

If Yes is selected, the operation will still be added to the sequence. If No is selected, the operation will not be added.

NOTE: This loop detection will only occur for the current event sequence being programmed. Warnings will not be displayed if other combinations of event/sequences are programmed that cause similar loops.

Dynamic Feature Support

Drivers may choose to dynamically add or remove support for commands, properties, and events, including those which are exposed to sequences. This is primarily intended to happen at configuration time since removing a command later may break programming that the user has configured. However, it is possible to change feature support at any time.

For information on dynamically adding or removing a feature, refer to Add Commands and Properties.
