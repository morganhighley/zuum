Create the Driver

The topics in this section describe how to create a platform driver. This tutorial contains incomplete examples that are used as a guide. For the complete sample code, refer to the Platform Sample driver provided in the Crestron Drivers SDK, which can be downloaded from Resources.

Creating a platform driver requires the following steps:

  1. Create the Driver Project
  2. Create the Driver Data File
  3. Implement the Main Driver Class (AGateway)
  4. Implement the Gateway Protocol (AGatewayProtocol)
  5. Create Drivers for Paired Devices
  6. Build the Driver PKG File
  7. Import the Driver into Crestron Home

Create the Driver Project

Create a new SIMPL# Pro library project in Visual Studio software as described in Create the Driver Project.

Create the Driver Data File

The driver data file is a JSON-formatted file that contains information about the device, and it is embedded into the DLL file. Since the data file format is the same across all device types, only some of the information in a data file applies to platform devices.

To create a data file for a platform device, modify the example given below with the details about your device.

NOTE: Platform drivers require the drivers for paired devices to be listed in the Dependencies section of the data file. Refer to Dependencies for more information.

After the data file is created, it will need to be imported into the driver’s project and set as an embedded resource.

To add an existing JSON file to the project:

  1. Right click on the project in Visual Studio.
  2. Navigate to Add > Existing Item.
  3. Select the JSON file.
  4. Select Add.

To set the JSON file as an embedded resource, navigate to the properties of the file in Visual Studio, and for Build Action, select Embedded Resource.

Implement the Main Driver Class (AGateway)

Create a new class that implements AGateway and the transport interface being used. In the Initialize method set the transport and the protocol. The protocol class will be created in the next step.

The following example code uses the ITcp transport interface.

Copy 
public class SampleGateway : AGateway, ITcp 
{ 
    public void Initialize(IPAddress ipAddress, int port) 
    { 
        var transport = new SampleGatewayTransport 
        { 
            EnableLogging = InternalEnableLogging, 
            CustomLogger = InternalCustomLogger, 
            EnableRxDebug = InternalEnableRxDebug, 
            EnableTxDebug = InternalEnableTxDebug 
        }; 

        ConnectionTransport = transport; 

        Protocol = new SampleGatewayProtocol(ConnectionTransport, Id) 
        { 
            EnableLogging = InternalEnableLogging, 
            CustomLogger = InternalCustomLogger 
        }; 
    }
} 

Implement the Gateway Protocol (AGatewayProtocol)

Create a new class that implements AGatewayProtocol. The gateway will manage its paired devices within this class. Refer to AGatewayProtocol Class for more information on the available methods.

Copy 
public class SampleGatewayProtocol : AGatewayProtocol 
{ 
    public SampleGatewayProtocol(ISerialTransport transport, byte id) : base(transport, id) 
    { 

    } 
} 


An example workflow of how to add a paired device to the gateway is provided below:

  1. Send a request to the physical gateway (a web API is used for this example).
  2. A response is received from the API saying that there is a new device associated (or paired) to the account.
  3. Use the information from that device to call the AddPairedDevice method in the AGatewayProtocol base class.

    4. Copy 
    private void AddPairedDeviceExample(IDeviceFromApi pairedDevice) 
    { 
        // Using the information received from the web api create an instance of the corresponding driver (ABasicDriver class) for this paired device 

        // Call AddPairedDevice in the base class to notify the gateway of the newly paired device 
        AddPairedDevice(*Insert GatewayPairedDeviceInformation for the device*, *Insert instance of the device (ABasicDriver)*); 
    } 

  4. Use the other methods such as RemovePairedDevice and UpdatePairedDevice as needed to keep the information in sync between the gateway driver and the web API.

When the platform device is removed from the system, the paired devices must be disposed. In the platform driver’s protocol implementation, the protocol should override the Dispose() method and dispose of all paired devices before calling base.Dispose(). This disposal is necessary to ensure that any paired device drivers created by the platform get cleaned up properly.

Refer to the sample code below or to the Platform Sample driver provided in the Crestron Drivers SDK, which can be downloaded from Resources.

Copy 
public override void Dispose()
{
     try
     {
           _pairedDevicesLock.Enter();

           foreach (var pairedDevice in _pairedDevices.Values)
           {
                if (pairedDevice is IDisposable)
                     ((IDisposable) pairedDevice).Dispose();
           }

           _pairedDevices.Clear();
     }
     finally
     {
           _pairedDevicesLock.Leave();
     }

     base.Dispose();
}

Create Drivers for Paired Devices

The paired devices must be created as separate SIMPL# Pro library projects but can be within the same solution. Each paired device must also have their own JSON data file.

NOTE: Paired devices can be of any supported device. If the device type is not a type currently supported by the Crestron Drivers SDK framework, a Crestron Home Extension device type must be used.

For platform drivers that contain Crestron Home Extension drivers, there will only be one IncludeInPkg folder with UiDefinitions and Translations subfolders that are shared between all Crestron Home Extension drivers. Each Crestron Home Extension driver must override the GetUiDefinition method in order to read in the correct UI definition for that driver.

Below is an example of a simple paired Crestron Home Extension driver that overrides GetUiDefinition and also contains an instance of GatewayPairedDeviceInformation that will be used to identify this device on the gateway:

NOTE: For the complete implementation, refer to the Platform Sample driver provided in the Crestron Drivers SDK, which can be downloaded from Resources. For more information on creating Crestron Home Extension drivers refer to Create the Driver.

Copy 
public class SamplePairedExtension : AExtensionDevice 
{ 
    private int _tapCount; 
    private readonly PropertyValue<int> _tapCountProperty; 
    private readonly GatewayPairedDeviceInformation _pairedDeviceInfo; 

    public SamplePairedExtension(string id, string name) 
    { 
        _tapCount = 0; 
        _tapCountProperty = CreateProperty<int>(new PropertyDefinition("TapCount", string.Empty, DevicePropertyType.Int32)); 

        _pairedDeviceInfo = new GatewayPairedDeviceInformation( 
            id, 
            name, 
            Description, 
            Manufacturer, 
            BaseModel, 
            DriverData.CrestronSerialDeviceApi.GeneralInformation.DeviceType, 
            string.Empty); 
    } 

    protected override string GetUiDefinition(string uiFolderPath) 
    { 
        var uiFilePath = Path.Combine(uiFolderPath, "PairedExtensionUi.xml"); 

        if (!File.Exists(uiFilePath)) 
        { 
            Log(string.Format("ERROR: Ui Definition file not found. Path: {0}", uiFilePath)); 
            return null; 
        } 

        if (EnableLogging) 
            Log(string.Format("UI Definition file found. Path: '{0}'", uiFilePath)); 

        return File.ReadToEnd(uiFilePath, Encoding.UTF8); 
    } 

    protected override IOperationResult DoCommand(string command, string[] parameters) 
    { 
        switch (command) 
        { 
            case "ButtonTap": 
                _tapCount++; 
                _tapCountProperty.Value = _tapCount; 
                Commit(); 
                break; 
        } 

        return new OperationResult(OperationResultCode.Success); 
    } 

    protected override IOperationResult SetDriverPropertyValue<T>(string propertyKey, T value) 
    { 
        return new OperationResult(OperationResultCode.Error, "this method is not implemented"); 
    } 

 

    protected override IOperationResult SetDriverPropertyValue<T>(string objectId, string propertyKey, T value) 

    { 

        return new OperationResult(OperationResultCode.Error, "this method is not implemented"); 

    } 

    public GatewayPairedDeviceInformation PairedDeviceInformation 
    { 
        get { return _pairedDeviceInfo; } 
    } 
} 


For a standard driver, the consuming application would create an instance of the driver class using reflection. For platform drivers, the consuming application expects to receive an instance of the paired devices from the driver. Therefore, your platform driver must instantiate each paired device as necessary.

To instantiate each paired device, your platform driver must reference each project for the paired devices. Once referenced, the platform can instantiate an instance of the driver as shown below:

Copy 
// Create an instance of the paired device 
var extensionPairedDevice = new SamplePairedExtension("ID", "Name"); 

// Add the paired device to the gateway 
AddPairedDevice(extensionPairedDevice.PairedDeviceInformation, extensionPairedDevice); 

In this example, the class SamplePairedExtension is a Crestron Home Extension driver that inherits from AExtensionDevice. Two arguments were added to the constructor to take in an ID and a name, which will be used by the GatewayPairedDeviceInformation class for the device. In the following line, the AddPairedDevice method is called to notify the gateway protocol of the newly added device. From there, the base implementation of the protocol class will automatically notify the consuming application of the added device and provide it with the instance of the driver.

Build the Driver PKG File

Once your driver code is complete, you must build it into a PKG file:

  1. Build the driver solution.
  2. Copy the DLL file and all dependencies (listed in Dependencies section of the data file) from the output to the folder containing the Manifest Utility tool. Refer to the README.txt file in this folder for additional information about this tool.
  3. If the platform contains any Crestron Home Extension devices, include the IncludeInPkg folder as well before running the Manifest Utility tool.
  4. Run the Manifest Utility tool by double clicking ManifestUtil.exe.

If the build is successful, a PKG file with the same name as the DLL file will be created.

NOTES:

  • Do not change the name of the PKG file. It must match the DLL file.
  • For platform drivers, a PKG file will also be created for each of the paired devices. These drivers are not functional without the platform, so they can be ignored. The only required PKG file is the one created for the platform itself, as it will contain all the DLL files for the paired device.

To verify the contents of a PKG file, it can be unzipped using 7-Zip or similar utility.

Platform devices should contain the following contents:

  • The DLL file for the platform driver
  • The DLL file for each of the dependencies listed in the Dependencies section of the data file
  • A DAT file containing a subset of the information from the JSON data file.
  • If the platform contains one or more Crestron Home Extension drivers as paired devices, then it should also contain:
    • A folder named uidefinitions containing the UI definition provided.
    • A folder named translations containing the translation files provided.

Import the Driver into Crestron Home

Import the platform driver into Crestron Home as described in refer to Side Load Device Drivers. Once imported, the driver should be available for selection as a third‑party device in the Crestron Home Setup app.
