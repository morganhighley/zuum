Entry API Reference

This section lists the various properties, commands, events, and types that make up the Device Driver Entity Model API for Entry devices.

Entry

entry

This capabilitity provides information related to an entry device (i.e. doors, gates, garage doors).

Types

State

Fully Qualified Id: entry:State

Lists the various states that an entry device can have.

The values in this enum are specific to the entry capability.

Definition
Copy 
[EntityDataType(Id = "entry:State")]
public enum EntryState
{
    /// <summary>
    /// Indicates the closed state.
    /// </summary>
    Closed,

    /// <summary>
    /// Indicates the fully open state.
    /// </summary>
    Open,

    /// <summary>
    /// Indicates the entry is neither fully closed nor fully open.
    /// Only use this value if the device is able to distinguish between partially open and
    /// fully open.
    /// If the device is not able to distinguish between partially open and fully open, use
    /// the "open" state.
    /// </summary>
    PartiallyOpen,

    /// <summary>
    /// Indicates the entry device is moving in the open direction.
    /// </summary>
    Opening,

    /// <summary>
    /// Indicates the entry device is moving in the closed direction.
    /// </summary>
    Closing,

    /// <summary>
    /// Indicates the entry device is moving and the direction is not known.
    /// If the direction of movement is known, use either the "opening" or "closing" value.
    /// </summary>
    Moving,
}
Error

Fully Qualified Id: entry:Error

Lists the various errors that an entry device can have.

Definition
Copy 
[EntityDataType(Id = "entry:Error")]
public enum EntryError
{
    /// <summary>
    /// Indicates the entry device is jammed and cannot move.
    /// </summary>
    Jammed,

    /// <summary>
    /// Indicates that an object was detected in the path of the entry device.
    /// </summary>
    ObjectInPath,

    /// <summary>
    /// Indicates the entry device was manually stopped.
    /// Once the entry device begins to move, this error should clear.
    /// </summary>
    ManuallyStopped,

    /// <summary>
    /// Indicates an unknown error has occurred.
    /// </summary>
    Unknown,
}

Commands

open

Fully Qualified Id: entry:open

Sends the open command to the device.

Driver Implementation Note: The open command should be sent to the device even if the device is already "Open".

This is required in order to prevent security risks due to incorrect, missing, or delayed feedback from the device.

    // TODO Your code here

toggle

Fully Qualified Id: entry:toggle

Sends the toggle command to the device.

The toggle command will close the entry if it is open, and open the entry if it is closed.

If the entry is partially open, the movement of the entry device will depend on the device implementation.

Example
Copy 
[EntityCommand(Id = "entry:toggle")]

{

}
holdOpen

Fully Qualified Id: entry:holdOpen

Sends the hold open command to the device.

Some entry devices automatically close after a certain amount of time (i.e. gates).

This command will open the device and have it remain open until the close or toggle command are called.

Driver Implementation Note: This command should only be implemented if the device automatically closes after a certain time period and the device supports the hold open feature.

systemComponentVersions

Fully Qualified Id: deviceInformation:systemComponentVersions

Collection of versions for the various system components

The keys of this collection must be user-friendly identifiers for what version this is. They typically match strings displayed on the device's interface and/or manual. firmwareVersion is used for the device's "primary" firmware version. That property should be used when there is an obvious corresponding value on the device. This property must not include the version from firmwareVersion as that may cause applications to report the version twice. Applications should not assume versions will not be added or removed, even at runtime. A device update may cause it to add, remove, or rename component versions.

typeDef 

    type: object

    itemType: string

Example
Copy 
[EntityProperty(Id = "deviceInformation:systemComponentVersions")]
public IDictionary<string, string> DeviceInformationSystemComponentVersions { get; set; }

private void TriggerDeviceInformationSystemComponentVersionsChangedEvent()
{
    NotifyPropertyChanged("deviceInformation:systemComponentVersions", CreateValueForEntries(DeviceInformationSystemComponentVersions));
}
additionalProperties

Fully Qualified Id: deviceInformation:additionalProperties

An array of names of properties that provide additional information about the device.

These properties are typically driver-defined and are specific to the driver or device, not ones defined in other capabilities. They are intended to allow a device to specify more information about itself than is described by the specific properties existent in this capability.

Note that these properties are typically for informational purposes only. They are intended to be shown on the user interface for the device. They should not be used to determine any type of functionality unless explicitly defined for other purposes.

deviceName

Capability to be implemented by a device or device component which supports a configurable name

This capability is reserved for devices which report their own user-configurable name, and potentially the ability to set that name via the API.

This capability should not be "faked" where the driver emulates this feature.

This "name" is for describing and identifying the device in some user interface. Sometimes the device supports configuring its host name on the network, but that is not the intent of this interface.

This is NOT the label associated with a device component. For that, see the deviceComponentLabel capability.

For example, a power controller may have multiple banks, labeled something like "Bank 2" or "Aux" on the power controller chassis. It may also support a software-defined "name" for the bank that may be changed via the device user interface. This capability would be used to report that software-defined name, not the "Bank 2" or "Aux" label.

Status

errorIndicator

Capability to be implemented by a device which supports reporting errors

Types

DeviceErrorCode

Fully Qualified Id: errorIndicator:DeviceErrorCode

Defines a device error code

These are unique per device only. Comparing error codes from different devices may yield incorrect results.

These specific error codes may be consumed by a user interface that is aware of a specific device model and its types of errors.

Properties
Name Details
code

Error code identifying the error that occurred

This is contextual to the manufacturer and/or the device model that raised the error. The manufacturer's documentation may be needed to decipher this error.

NOTE: This is a string because some manufacturers may use alphanumeric error codes.

Type: string
message

Error message describing the error code

Typically this string will be in English so that Crestron can diagnose and debug the error, but depending on the driver's author or device's language, it may be in their native language.

Type: string
severity

Indicates the severity of the error

Type: ErrorIndicatorDeviceErrorCodeSeverity

readyIndicator

Capability to be implemented by a device which indicates when it is "ready" to accept commands. This is not necessarily the same as an "online" flag. The device may be online, but not "ready" yet.

Properties

isReady

Fully Qualified Id: readyIndicator:isReady

True when the device is ready to receive commands and return current information via any properties or commands with return values, as well as raise (feedback) events (if supported).

When this is false, it typically means the device is either offline or just not ready to communicate yet (i.e. online, but still loading).

Unless otherwise noted in a specific device capability property, command or event, all interactions with the device will not be available or valid until this property turns true.

Methods and properties may still be called while this property is false, however their values will be deemed unreliable (and will be set to either their last value or the default value) until this property changes to true.

If an error can occur that will prevent the device from ever being ready, the errorIndicator capability should be implemented in order to raise the error.

The device may also become ready, but have its errorIndicator:errorCodes property already set. This means the device is functional, but has errors that may prevent it from functioning properly.

All other errors will be raised after the device is ready. If the device stays ready, it means the device is functional, but has errors that may prevent it from functioning properly. If the device changes to not ready, it means the error is causing the device to be non-functional.

typeDef 

    type: boolean

Example
Copy 
[EntityProperty(Id = "readyIndicator:isReady")]
public bool ReadyIndicatorIsReady { get; set; }

private void TriggerReadyIndicatorIsReadyChangedEvent()
{
    NotifyPropertyChanged("readyIndicator:isReady", new DriverEntityValue(ReadyIndicatorIsReady));
}
