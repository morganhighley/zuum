Multizone AV Receiver Drivers

This topic describes how to create a Crestron Driver for a multizone AV receiver device. Multizone AV receivers allow an AV receiver to send more than one signal to speakers or a separate audio system in different locations.

Device Interfaces

The following device interfaces are provided for multizone AV receiver drivers.

IBasicAvr

Refer to the following diagram for an overview of the IBasicAvr interface.

IAvrZone

AVR Zones are a property of IBasicAvr as a List<IAvrZone> AvrZones. Each zone will be self‑contained and will work independently. The primary AVR control should have no impact on the remote zones.

The AVR Zone utilizes many of the same features that the Main AVR uses:

  • Power
  • Volume
  • Mute
  • Input Select
  • Audio Input Select

The AVR Zone also includes many support properties that can be used to enable or disable UI objects in the program as needed to support the specific device.

The properties include all supported features for the zone. This is used to enable or disable any user interface items needed to control the zone as well as to provide a way to update any status.

Supported Features are as follows:

  • SupportsAudioInputFeedback
  • SupportsChangeVolume
  • SupportsDiscreteMute
  • SupportsDiscretePower
  • SupportsMute
  • SupportsMuteFeedback
  • SupportsPowerFeedback
  • SupportsAudioInputSource
  • SupportsVideoInputSource
  • SupportsSetVolume
  • SupportsTogglePower
  • SupportsVideoInputFeedback
  • SupportsVolumePercentFeedback

Additional properties used to get and set status of the current zone are as follows:

  • AudioInOut
  • AudioInputIs
  • AudioInputSource
  • CoolingDown
  • FriendlyName
  • IsMuted
  • PowerIsOn
  • Uid
  • VideoInOut
  • VideoInputIs
  • VideoInputSource
  • VolumePercent
  • WarmingUp
  • ZonesExist

IAvrZone Methods

The AVR zone will have many of the same methods used in the Main AVR. They are implemented in the same way and are independent from the Main AVR. Refer to the following diagram.

IAvrZone Events

Event handling is also contained within the remote zone. Each zone will fire events based on feedback and driver input. These events can also be registered and watched from within the main AVR driver if needed.

New Abstract Classes

The following new abstract classes are provided for multizone AV receiver drivers.

ABasicAVReceiver

The ABasicAVReceiver class inherits from IBasicAVR and ABasicDriver. This contains all the methods and properties defined by IBasicAVR that are not shown. Additional fields added to support AVR Zones are as follows:

  • AvrZoneData: Data for all zones associated with the main driver. This data must be included in the main driver json.
  • The current framework inherits from IBasicAvr7, which adds the AvrZoneData field to AvrData.

AVReceiverProtocol

The AVReceiverProtocol class inherits from ABaseDriverProtocol.

NOTE: All methods that are available on IBasicAvr and implemented here, but the names of the methods may differ. These methods were not included; only methods that are not available in any interface are present here. The AvrZoneData is included here within AvrData and gives access to the driver developer to override and of the commands within the zone. This is done by overriding specific commands included within the protocol class. All of these methods have the word “Override” within their names and require a reference to the zone being controlled as an argument in the method signature.

The following methods are used when response processing within the framework does not meet expectation and requires some additional processing for the specific device. All method return types are bools with “true” return disabling the base processing.

Methods that can be overridden are as follows:

  • bool OverrideDeconstructZoneAudioInput(AAvrZone zone, string response)
  • bool OverrideDeconstructZoneInput(AAvrZone zone, string response)
  • bool OverrideDeconstructZoneMute(AAvrZone zone, string response)
  • bool OverrideDeconstructZonePower(AAvrZone zone, string response)
  • bool OverrideDeconstructZoneVolume(AAvrZone zone, string response)

The control method overrides use the same model allowing the developer to disable the base method calls by returning “true” withing these calls.

  • bool OverrideZoneDecrementVolume(CommonCommandGroupType zone)
  • bool OverrideZoneIncrementVolume(CommonCommandGroupType zone)
  • bool OverrideZoneMute(CommonCommandGroupType zone)
  • bool OverrideZoneMuteOn(CommonCommandGroupType zone)
  • bool OverrideZoneMuteOff(CommonCommandGroupType zone)
  • bool OverrideZonePowerToggle(CommonCommandGroupType zone)
  • bool OverrideZonePowerOn(CommonCommandGroupType zone)
  • bool OverrideZonePowerOff(CommonCommandGroupType zone)
  • bool OverrideZoneSetVideoInputSource(CommonCommandGroupType zone, VideoConnections input)
  • bool OverrideZoneSetAudioInputSource(CommonCommandGroupType zone, AudioConnections input)
  • bool OverrideZoneSetVolume(CommonCommandGroupType zone, uint volume)
  • bool OverrideZoneVolumeDown(CommonCommandGroupType zone, CommandAction action)
  • bool OverrideZoneVolumeUp(CommonCommandGroupType zone, CommandAction action)

Driver JSON (Zone Objects)

The driver JSON file can support up to five additional zones. While using five zones is very rare, this framework was set up to be able to include this number in the event that it becomes a possibility. Additionally, each remote zone can also support video as well as audio. Typically, the remote zones would support audio only sources, but some newer models have been including video outputs for some extended zones.

It is important to note how drivers are used with Crestron products that consume them. When creating a multizone driver, it is important that the following rules are followed in order to allow for Crestron Home or AV Framework™ software to use the drivers.

  • When defining inputs, use the standard commands that are more generic rather than the very specific (HDMI, DVI, Network, and so forth).
    • It is ideal that Standard Command values are “Input1” to “InputN”.
    • The above permits the developer to define which inputs will be available on these platforms and allows them to be defined as either Audio, Video or Both.
  • The order and enum values in the main zone must also be followed in the remote zone definitions.
    • Crestron Home does not provide separate pages for the different zones. All of the routing takes place on a single page so the input mapping must match between Main and the Remote Zones.
    • Defining the inputs this way will also ensure that all of the sources that you want to have available in each zone can be used by the customer.
