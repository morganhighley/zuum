Light API Reference

This section lists the various properties, commands, events, and types that make up the Device Driver Entity Model API for Light devices.

Common

commandFunction

Defines an enumeration to be used for press and hold functionality across many driver capabilities

Types

crestron:CommandFunction

Fully Qualified Id: crestron:CommandFunction

Describes the operation of a command which may be issued once or may be repeated for some duration.

The most common use case for this is remote control button emulation. A single tap would send a pulse, but pressing and holding the button would send high until the button is released and then low at the end.

Sending high and then low immediately is not equivalent to sending pulse in all cases. Applications should make their own determination on when to treat a press as a pulse versus high/low.

Definition
Copy 
[EntityDataType(Id = "crestron:CommandFunction")]
public enum CommandFunction
{
    /// <summary>
    /// Indicates a single, momentary operation.
    /// If <see cref="High"/> was previously used, this stops that operation and begins the
    /// new single pulse operation.
    /// </summary>
    Pulse,

    /// <summary>
    /// Indicates the beginning of an ongoing operation.
    /// If <see cref="High"/> was previously used, this will have no effect since it is
    /// already started.
    /// Callers MUST ensure they follow up with <see cref="Low"/> eventually to avoid a
    /// runaway condition where the operation goes on forever.
    /// </summary>
    High,

    /// <summary>
    /// Indicates the end of an ongoing operation.
    /// If <see cref="High"/> was not previously used, this will have no effect since it is
    /// already stopped.
    /// </summary>
    Low,
}

levelTransition

Defines a type used for levels that transition over a period of time

Types

crestron:LevelTransition

Fully Qualified Id: crestron:LevelTransition

Describes a transition from one value to another

When paired with a property indicating the present level, that property will typically report a stale value during a transition. The property with this type will typically be null when a transition is not active. However, refer to the documentation of the specific properties for final clarification.

Properties
Name Details
target

The target that the level will be at the end of the transition period

Note that if the transition is interrupted, the actual level is not guaranteed to reach this value.

Type: double
duration

The total duration of the transition from start to target

If omitted or zero, the duration is unknown. This duration should not be updated during the transition since it represents the total transition time, not remaining time.

If the application needs to animate the transition, it must listen for property changes before this property updates in order to be notified and know when the transition starts.

Type: long
start

The starting value being transitioned from, if known

May be omitted if not known. Note that a driver that does report the starting value once may still later report a transition without a start value.

Type: double?

Fully Qualified Id: crestron:DeviceUxCategory

The User Experience category for a device

This describes the type of device that it is. It should ONLY be used for categorization purposes for selecting an icon or helping organize the device into a list. It should NOT be used to determine how to use the device.

In practice, devices can fit into multiple of these categories. The driver developer may have to choose what they consider the closest, or use "other" if no other categories apply. Applications should inspect the definition of the device instance to determine its capabilities and assume nothing about the device based on this string.

Identification

deviceComponentLabel

Capability to be implemented by a device or device component which is labeled

This label is typically affixed onto the device itself. The label must never change as this is intended to represent the physical marking on the device. For labels that may change dynamically, consider if the deviceName capability applies.

This capability is primarily implemented by sub-devices which represent components of the main device.

For example, a power controller may have multiple banks, labeled something like "Bank 2" or "Aux" on the power controller chassis. It may also support a software-defined "name" for the bank that may be changed via the device user interface. This capability would be used to report that "Bank 2" or "Aux" label that is physically attached to the device (not the software-defined "name").

deviceName

Capability to be implemented by a device or device component which supports a configurable name

This capability is reserved for devices which report their own user-configurable name, and potentially the ability to set that name via the API.

This capability should not be "faked" where the driver emulates this feature.

This "name" is for describing and identifying the device in some user interface. Sometimes the device supports configuring its host name on the network, but that is not the intent of this interface.

This is NOT the label associated with a device component. For that, see the deviceComponentLabel capability.

For example, a power controller may have multiple banks, labeled something like "Bank 2" or "Aux" on the power controller chassis. It may also support a software-defined "name" for the bank that may be changed via the device user interface. This capability would be used to report that software-defined name, not the "Bank 2" or "Aux" label.

lightColor

Enables control of the hue and saturation of a light

Used in conjunction with the lightDimmer capability to control the hue, saturation, and level of the light.

May interact with lightColorTemperature or lightEmulatedColorTemperature as well.

Also see the lightTunable:setLevels command which provides a way to control the color in sync with other parameters.

Commands

setSaturation

Fully Qualified Id: lightColor:setSaturation

Sets the saturation level of the light

In combination with hue, controls the color of the light.

If the light is off and the device does not support changing saturation while keeping the light off (such that it is applied whenever it later turns on), the driver should cache the value and apply it when the light does eventually turn on. Applications should note that this may not always be possible in some device APIs and it is possible that changing the saturation ends up turning the light on, although it is preferable not to.

If the application wants to ensure consistency across all devices, it may want to turn the light on before executing this command.

lightColorTemperature

Enables control of the color temperature of a light

Use this capability to control a light which does not support hue or saturation channels, or a light where the color temperature operates independently of hue and adjusting saturation mixes hue and color temperature.

In other words, use this when saturation does not exist, or when 0.0 saturation means entirely controlled by color temperature, and 1.0 saturation means color temperature has no effect, and 0.5 saturation means color temperature and hue are mixed to obtain the final color.

When setting color temperature is really just setting a calculated hue and saturation value, do not use this capability. Use lightEmulatedColorTemperature instead.

Also see the lightTunable:setLevels command which provides a way to control the color temperature in sync with other parameters.

lightEmulatedColorTemperature

Enables control of the emulated color temperature of a light

Use this capability to control a light which supports hue and saturation but also includes an API to control the color temperature which overrides the hue and saturation.

In contrast with the lightColorTemperature capability which operates in combination with hue and saturation, this capability supports setting color temperature instead of it. The device will internally calculate the appropriate hue and saturation values, and changing color temperature will typically result in an update to reported hue and saturation values.

If the device only supports color temperature and not hue and saturation, the lightColorTemperature capability should be preferred instead of of this one. However, applications must support this capability being implemented even when color control is not supported. That way, drivers for a family of devices which use emulated color temperature can use a consistent implementation even for bulbs in the family that are color-temperature tuning only.

If the device does not support color temperature via its API or document an official conversion formula for the specific device, the driver should not "fake" this feature by calculating color temperature itself using a generic formula as that may not create an accurate color temperature representation due to variations in color accuracy of the light. If an application wants to support color temperature on a light by calculating hue and saturation values, it is perfectly capable of doing that itself.

Also see the lightTunable:setLevels command which provides a way to control the emulated color temperature in sync with other parameters.

Capability to be implemented by a device which supports "scenes" which may apply to this device or to multiple sub-devices.

More than one scene may be active at a given time as some scenes may contain other scenes, and other scenes might not overlap.
