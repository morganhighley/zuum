User Interface Files

The following topics describe files that must be created when working with the Crestron Home™ OS user interface.

UI Definition

The UI Definition is an XML file containing the tiles, layouts, and alerts that are rendered on the user interface.  It defines what controls are available as well as how those controls interact with the driver.

The UI Definition file is composed of the following sections:

Name Description
Version Indicates the version of the UI Definition file.  The current version is “2.0”.  This version may change as new features are added.  
Tile Defines how the home/room tile will appear and function.
Layouts Contains the collection of layouts for this definition.  Each layout will contain the set of controls that will be available for that layout.
Alerts Contains the set of alerts that are available to all layouts.


When creating the XML file for the UI Definition, it is recommended to use an XML editor which supports schema validation (such as Visual Studio Code with the Red Hat® XML extension).

NOTE: The UI definition must be encoded using UTF-8. Other encodings are not supported. Also, ensure that the UI definition does not contain a BOM at the beginning of the file. If it contains a BOM, Visual Studio Code will show your encoding at the bottom as “UTF-8 with BOM”. To change it, click the encoding, select Save with encoding, and select UTF-8.

The XML Schema Definition (XSD) can be found at the following URL: https://prd-use-rad-assets.azurewebsites.net/ExtensionsSchemaDefinition.xsd.

Refer to the sample code below:

<?xml version="1.0" encoding="UTF-8" ?>
<uidefinition xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:noNamespaceSchemaLocation="https://prd-use-rad-assets.azurewebsites.net/ExtensionsSchemaDefinition.xsd">
<version ver="2.0" />
<tile icon="#icSpa" navigation="show:main" showinhomepage="#false" showinroompage="#true" />
<layouts >
<layout id="main" isdefaultlayout="#true" >
<controls>
<toggle id="maintoggle" label="^power" value="{powerstate}" /></controls></layout></layouts>
<alerts /></uidefinition>

Tiles

An Extension device driver may define a tile that will appear on the home screen or on the room page (or both). This setting is defined in the UI definition. The tile definition in the UI Definition XML file specifies how the tile will be rendered.

Home Tile

Room Tiles

Sample XML:

<tile icon="#icSpa" status="{devicestatus}" navigation="show:main" action="command:togglepower" secondaryicon="{statusicon}" showinhomepage="#false" showinroompage="#true"/>

The tile element is composed of the attributes below. Optional attributes that are not used should be excluded from the XML tag, as assigning empty strings to unused attributes may result in errors.

NOTE: Any attributes with asterisks next to them in this section and the sections that follow are mandatory to implement.

Feature XML Attribute Value Type Binding Home Tile Room Tile Notes
Icon icon string Fixed icon, property Yes Yes The is a defined collection of available icons to choose from.

Examples:

Fixed icon: icon=“#iconName”

Dynamic icon: Icon=“{propertyName}"

See Bindings for more details on the syntax.
Secondary Icon secondaryicon string Fixed icon, property No Yes This icon currently shows on the bottom right of the room tile and follows the same syntax as the icon attribute.
Alternate Icon Text alticontext string Translation, property No Yes This field can be used to show a short string instead of an icon at the top left of the tile. If using this field, the “icon” attribute should be excluded.

Examples:

Translation: alticontext=“^translationKey”

Property: alticontext= “{propertyName}”
Status status string Translation, property Yes Yes The status shows underneath the title.

Examples:

Translation: status=“^translationKey”

Property: alticontext= status=“{propertyName}”
Action action string Action No Yes A command that will be executed when tapping on the tile.  If Navigation is defined, this field is optional.
Navigation navigation string Navigation Yes Yes Contains the id of the page that will be loaded when selecting the tile. If the tile has both action and navigation defined, an ellipsis (…) will be shown on the top right of the tile.

Tapping on the ellipsis will perform the navigation. If only navigation is defined, tapping on the tile will perform the navigation.
Title N/Ap N/Ap N/Ap Yes Yes The title of the tile. This field is the name that was assigned to the device when it was added to the system.
Display in Home Page* showinhomepage boolean Fixed Boolean Yes Yes Indicates whether the tile will be visible on the home page.

Generally, this would be set to “#false” as device tiles are visible in the room page by default.  Exceptions are made based on the device type and usage. 

Example:

showinhomepage:“#false”
Display in Room Page* showinroompage boolean Fixed Boolean Yes Yes Indicates whether the tile will be visible on the room page. This is set to “#true” in most cases.

Example:

showinroompage:“#true”

Layouts

Layouts contain the collection of pages and control objects for the user interface.  This is a grouping object that contains the collection of <layout /> elements.  There is no limit on how many layouts can be defined.

Sample XML:

<layouts>
<!-- layout elements would be inserted here--></layouts>

Layout

A layout is a page consisting of various control objects.  Each layout has a unique ID that is defined by the programmer.  This ID is used for navigating between layouts.

Layout Examples

Sample XML:

<layout id="settings" isdefaultlayout="#false" title="^name" subtitle="{devicestatus}" inputparameters="$userid">
<!-- controls group would be inserted here--></layout>

The layout attribute is composed of the attributes below.

XML Attribute Value Type Binding Notes
id* string N/Ap The ID of the layout attribute. This value must be unique.
title string Translation, property A title that is shown at the top of the page.
subtitle string Translation, property A subtitle that will be displayed underneath the title.
inputparameters string Variable Allows the layout to receive parameter(s) from another layout. For multiple parameters, use a comma between variables (“$variable1,$variable2”)

If the layout is marked as the default layout (isdefaultlayout=”#true”), then inputparameters are not allowed.

Both primitive types and objects may be assigned to a variable. If an object is assigned, controls may access the object subproperties using the dot (.) operator on the variable (e.g., $user.name).

For example, Layout1 has list of users (in a listbutton control). Selecting a user from the list will navigate to Layout2. The listbutton control can use the parameters attribute to pass information about which user was selected to Layout2 using the parameters attribute on the listbutton control. Layout2 will receive the information and store it in the variable defined in inputparameters.
isdefaultlayout boolean Fixed Boolean Defines which layout is the primary layout that the user interface should navigate to by default. The default layout does not allow for inputparameters.

There must be one layout in the UI definition that has this attribute as “#true”.

Alerts

Alerts define the collection of available alerts for the UI definition. Alerts can be displayed by targeting the alert with a control's navigation attribute (for example, navigation="showalert:alertId"). Refer to "navigation" in the table below for more information.

Sample XML:

<alerts>
<!-- alert elements would be inserted here--></alerts>

Alert Confirmation

An alert confirmation shows a dialog box when an alert is raised. The alert confirmation contains an alert message, a proceed button, and a cancel button.

Alert Confirmation Example

Sample XML:

<confirmation id="confirmationAlertID" messagelabel="^confirmationAlertMessage" cancelbuttonlabel="^confirmationCancelLabel" proceedbuttonlabel="^confirmationProceedLabel" proceedaction="command:confirm" actionparameters="$selectedObject.Id" navigation="show:settings" navigationparameters="$selectedObject" inputparameters="$selectedObject"/>

The alert confirmation attribute is composed of the attributes below.

XML Attribute Value Type Binding Notes
id* string N/Ap The ID of the alert confirmation attribute. This value must be unique.
messagelabel* string Translation, property The message that is shown in the alert confirmation.
proceedbuttonlabel* string Translation, property The text that is shown for the proceed button label.
cancelbuttonlabel* string Translation, property The text that is shown for the cancel button label.
proceedaction* string Command The command that will be sent to the driver when the user presses the proceed button.
navigation string Navigation Allows for a layout or alert to be shown when the user presses the proceed button. If navigation parameters are specified, they will be passed to the requested layout as the “inputparameters”.
navigationparameters string Property, variable Specifies which properties and/or variables will be passed to the requested layout as part of the navigation action. Multiple values can be sent by using a comma between each value.
inputparameters string Variable Allows the alert to receive a parameter(s) from another layout/alert. For multiple parameters, use a comma between variables (ie. “$variable1,$variable2”) Both primitive types and objects may be assigned to a variable.

If an object is assigned, controls may access the objects subproperties using the dot (.) operator on the variable (ie. $user.name).

Action Selector

This alert can be used to display a dialog box where multiple choices are available. Additional choices can be added by including additional button elements as children to the ActionSelector element. The button elements can perform action and/or navigation commands. The cancel button will always close the dialog and is not configurable.

Action Selector Example

Sample XML:

<actionselector id="actionSelectorID" title="^actionSelectorTitle"><button id="actionSelectorButton1ID" label="^actionSelectorButton1" action="command:action1"/>
<button id="actionSelectorButton2ID" label="^actionSelectorButton2" action="command:action2"/>
<button id="actionSelectorButton3ID" label="^actionSelectorButton3" action="command:action3"/></actionselector>

Translation Files

The device driver provides a translation dictionary for each of its supported locales. When the Crestron Home app connects to the Crestron Home system, it will provide the current culture information to Crestron Home, and Crestron Home will return the translation dictionary that matches the culture information (or the default if the requested dictionary does not exist).

Translation entries can be fixed text or they can have parameters (denoted by %s in the translation string).

Refer to the following example scenarios:

Example 1: German translation, no parameters

Key: “washing”, Value: “Waschen”

This is the typical translation lookup scenario where value provides the localized string.

 

Example 2: English translation with a parameter

Key: “errMsg1”, Value: “Unable to process request. Current Status: %s”

In this example, the %s will be replaced by a string that is defined at runtime. The string may be another translation lookup, or it may be statically defined text from the driver. See Bindings for more details.
