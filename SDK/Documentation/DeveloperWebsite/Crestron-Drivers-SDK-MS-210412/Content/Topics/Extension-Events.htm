Crestron Home Extension Driver Events

All Crestron Home Extension driver devices with events are displayed on the Actions & Events page for the room alongside other types of programmable devices and are ordered alphabetically by device name. The manufacturer name of the Crestron Home Extension device is also displayed.

If a Crestron Home Extension device does not have any events, it will not appear in this list. Selecting the device will display all available events that can be programmed.

The Sequence mode is available for programming use.

NOTE:  A future release of Crestron Home will provide the ability to program other modes such as Scene and Quick Actions.

Once the Sequence mode is chosen, a user experience is provided similar to other areas in the application.

If an Crestron Home Extension device is deleted or the room in which the Crestron Home Extension device is deleted, all programming associated with the extension device’s events will be deleted.

Programming Attributes

The following programming attributes are provided for Crestron Home Extension device events.

ProgrammableEventAttribute

To have an event for a Crestron Home Extension device appear on the Actions & Events page automatically, the event should be decorated with the ProgrammableEventAttribute.

Copy 
public class DoorLock
{
    [ProgrammableEvent]
    public event Action Locked;

    [ProgrammableEvent]
    public event Action Unlocked;
}


The name of the event will be used by default as the name of the event on the Actions & Events page. In the image below, notice that the Locked and Unlocked event names match the event names in the Crestron Home Extension driver code above.

To control the event name, the displayName parameter can be used. The following sample code changes the event name on the Actions & Events page to “Door Locked” even though the event name “Locked” was used.

Copy 
public class DoorLock
{
    [ProgrammableEvent ("Door Locked")]
    public event Action Locked;

    [ProgrammableEvent]
    public event Action Unlocked;
}


To localize the event name, the displayName parameter can be prepended with a “^” symbol. This practice is consistent with other strings that are localized in Crestron Home Extension devices.

Copy 
public class DoorLock
{
    [ProgrammableEvent ("^Door Locked")]
    public event Action Locked;

    [ProgrammableEvent]
    public event Action Unlocked;
}


When the Actions & Events page encounters a string that needs to be localized, the Crestron Home Extension device will be queried (the IRadExtensionDevice.GetTranslations method is called) for the translation string at runtime, and the localized string is displayed for the applicable item.

If the localized string is not found, the string will be displayed as is with the “^” symbol as shown in the image below.

NOTE: The ability to interpret event parameters is not supported at this time.

TriggeredByAttribute

Sequences that are triggered from events increase the chances of creating loops in the system. For example, the Locked event can be programmed to unlock a door lock and then the Unlocked event can be programmed to lock the same door lock.

To help reduce the risk of loops, warning messages will be displayed as applicable when operations are added to a sequence that target the same device as the event being programmed. These warning messages will be driven by the operation names specified in the TriggeredByAttribute.

Refer to the following sample code.

Copy 
public class DoorLock
{
    [ProgrammableEvent ("^Door Locked")]
    [TriggeredBy("Lock", "LockWithTimeout")]
    public event Action Locked;

    [ProgrammableEvent]
    public event Action Unlocked;

    [ProgrammableOperation]
    public void Lock()
    {

    }

    [ProgrammableOperation]
    public void Unlock()
    {

    }
}


In the code above, the TriggeredByAttribute is specifying that the Locked event is triggered by the Lock and LockWithTimeout operations. If the Lock or LockWithTimeout operations are added to the Locked event sequence, the following warning message will be displayed.

If Yes is selected, the operation will still be added to the sequence. If No is selected, the operation will not be added.

NOTE: This loop detection will only occur for the current event sequence being programmed. Warnings will not be displayed if other combinations of event/sequences are programmed that cause similar loops.

Exceptions will be thrown in the Manifest Utility tool during package generation if the operations specified in the TriggeredByAttribute are not found. All operations must be decorated with the ProgrammableOperationAttribute.

IsSupportedAttribute

The existing IsSupportedAttribute has been updated so it can be used to decorate events when an event should only exist under certain business conditions. The same semantics of the typeName, operationName, and optional path properties apply here as they did previously.

Before an event is displayed on the Actions & Events page, the method or property specified in the IsSupportedAttribute will be called. If the event is not supported (the method or property called returns false), the event will not be displayed on the Actions & Events page.

Copy 
public class DoorLock
{
    [ProgrammableEvent ("^Door Locked")]
    [IsSupported("DoorLock", "LockSupported", OperationType.Property)]
    public event Action Locked;

    [ProgrammableEvent]
    public event Action Unlocked;

    [ProgrammableOperation]
    public void Lock()
    {

    }

    public bool LockSupported
    {
        get
        {
            //Business logic determining whether or not the lock operation is supported goes here//
            return true;
        }
    }
}


Exceptions will be thrown in the Manifest Utility tool during package generation if the methods or properties specified in the IsSupportedAttribute are not found.
