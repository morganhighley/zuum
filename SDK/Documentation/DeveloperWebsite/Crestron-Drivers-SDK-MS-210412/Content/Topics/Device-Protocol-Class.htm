Device Protocol Class File

The device protocol class file contains the implementation of the device protocol class. This class controls how commands are sent, how feedback is parsed, and polling parameters.

The example driver API has the following properties:

  • All commands are sent in a packet with a header byte, the command bytes, and a delimiter byte.
    • Header = 0x0A
    • Delimiter = 0x0D
  • All feedback received will be sent in a packet with a feedback header byte, the type of feedback byte, the feedback byte(s), and a delimiter byte.
    • Feedback Header = 0x0B
    • Feedback type:
      • Power = 0x70
      • Input = 0x71
      • Volume = 0x72
    • Delimiter = 0x0D

NOTE: Since the API contains headers, some methods will need to be overridden since these parts are not defined in the Data File Helper.

In this topic, the TestManufacturerTestModelProtocol.cs file will be used as an example to implement the ADisplayProtocol class.

To create a new device protocol class file:

  1. Add a class to the project named TestManufacturerTestModelProtocol.cs.
  2. Define the namespace as ExampleDriver.
  3. Name the class TestManufacturerTestModelProtocol.
  4. Extend ADisplayProtocol.
  5. Add another class to this file within the namespace but outside of TestManufacturerTestModelProtocol, and name it ResponseValidator.
  6. Extend ResponseValidation with the ResponseValidator class. Then, create a constructor for this that takes an object of type DataValidation.
  7. Within ValidateResponse, remove any data that was not defined in the Data File Helper. Then, you can either return the new response using a base call or determine what type of feedback this is manually, whether or not it is complete, and which DeConstruct() method should be called.

    8. The base class will do the following:

    • Checks for the ID parameter (!$[id]) within the feedback header and replace it with the ID
    • If the main feedback header is present, strips it out of the response.
    • Checks the packet against the Ack/Nak definitions
    • Checks the packet against all feedback headers (power, input, volume, and so forth) and removes the one that is found from the response. It will assign CommandGroup based on this, set Data to the stripped response, and set Ready to true.

    This driver will need to override ValidateResponse because there is a delimiter that will need to be removed at the end of the packet.

  8. Within the overridden ValidateResponse, check to see if the delimiter exists:
    • If it doesnâ€™t, return a new instance of ValidatedRxData with Ready set to false and the response set to an empty string.
    • If it does, remove the delimiter from the response and return the base call.
  9. Within the TestManufacturerTestModelProtocol class created in Step 3, create a constructor that takes two parameters: ISerialTransport and byte.
  10. Within this constructor, assign ResponseValidation with a new instance of ResponseValidator that was created in Step 6. It will require one parameter (ValidatedData).
  11. Define the polling sequence for Power On and Power off using an array of StandardCommandsEnums. These sequences are located within ValidatedData.
  12. Override PrepareStringThenSend, since the header byte and delimiter byte will be appended to each command prior to sending

    13. In this example, the developer could have included this information within each command in the Data File Helper but chose to append it at run time.

  13. Within the overridden PrepareStringThenSend, append the header and delimiter to the command and return with a base call.

There are many methods that can be overridden based on the needs of the driver. Certain drivers will need to override SetVolume() if they need to format the volume level a specific way. The process for response validation can be done a number of ways; this example only shows one way of doing it. The currently released drivers all show several unique ways to parse a response and some example of overridden methods.

NOTES: 

  • When a particular packet needs to be ignored, return an instance of ValidatedRxData with the Ignore property set to true.
  • PowerOffPollingSequence will only execute Power commands

The following sample code shows the implementation of the procedure above.

using System;
using Crestron.Display;
using Crestron.ThirdPartyCommon.Transports;
using Crestron.ThirdPartyCommon.StandardCommands;

namespace ExampleDriver
{
public class TestManufacturerTestModelProtocol : ADisplayProtocol
{
public TestManufacturerTestModelProtocol(ISerialTransport transport, byte ID)
: base(transport, ID)
{
//Assign our response validator
ResponseValidation = new ResponseValidator(ValidatedData);

//Define the power on/off polling sequences
ValidatedData.PowerOnPollingSequence = new[] { StandardCommandsEnum.PowerPoll, StandardCommandsEnum.VolumePoll, StandardCommandsEnum.InputPoll };

ValidatedData.PowerOffPollingSequence = new[] { StandardCommandsEnum.PowerPoll };

//Define the polling interval (ms)
PollingInterval = 1000;
}

protected override void PrepareStringThenSend(ADisplayProtocol.CommandSet commandSet)
{
//Append the header byte and delimiter byte to the command
commandSet.Command = "\u000A" + commandSet.Command + "\u000D";

//Return the altered command
base.PrepareStringThenSend(commandSet);}}

public class ResponseValidator : ResponseValidation
{
public const string Delimiter = "\u000D";

public ResponseValidator(DataValidation DataValidation) : base (DataValidation)
{ }

public override ValidatedRxData ValidateResponse(string response, SupportedCommandGroups commandGroup)
{
ValidatedRxData validatedData = new ValidatedRxData(false, string.Empty);

//Check for delimter
if (!response.Contains(Delimiter))
{
return validatedData;}

//Remove delimiter
response = response.Substring(response.IndexOf(Delimiter) + Delimiter.Length, response.Length - Delimiter.Length);

//Return with base call
return base.ValidateResponse(response, commandGroup);}}}
